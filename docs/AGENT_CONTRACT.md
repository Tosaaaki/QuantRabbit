## 2. Operator Model

The trader is a single role with one playbook (`docs/SKILL_trader.md`). The scheduled task picks which model executes a given cycle; the playbook, the gateways, and the broker account are identical regardless of which model runs. In the active Codex workflow, “AI trader” and “GPT” name this same scheduled reasoning role; deterministic bots monitor, calculate, validate, and execute its bounded decision, rather than acting as a second AI decision-maker. There is no "Codex trader" vs "Claude trader" — there is **the trader**, and a scheduler that dispatches it.

Rules:

- **Exactly one trader scheduled task enabled at a time.** Two tasks against the same OANDA account = duplicate orders, double-counted exposure, contradictory protection actions.
- **DecaBot is separate from the main trader scheduler.** DecaBot lives at `/Users/tossaki/App/DecaBot`, uses its own OANDA account and launchd agents, and is operated from this repo only through the operator bridge `scripts/qr-decabot.sh` plus the weekend switcher's launchd pause/restore of DecaBot labels. It must not be counted as a second QuantRabbit scheduler against the main account, and QuantRabbit live code must not import or route DecaBot order writes.
- **Switch by enabling / disabling scheduled tasks.** No code, prompt, or playbook changes are required to swap the executing model.
- **Weekend scheduler guard.** Codex automation `qr-weekend-market-off` checks Saturday 06:00 and 07:00 JST and pauses related QuantRabbit schedulers only at the first DST-aware Friday 17:00 America/New_York weekly close; `qr-weekend-market-on` checks Monday 06:00 and 07:00 JST and restores them only at the corresponding Sunday 17:00 New York weekly open (06:00 JST in New York summer time, 07:00 JST in winter). Pre-boundary checks preserve the current task state without creating or consuming the weekend snapshot. The switcher snapshots current scheduler state only when the market has closed and restores only that snapshot, so a disabled Claude trader is not enabled accidentally and multiple trader schedulers are never restored. By default it manages weekday QuantRabbit Codex automations discovered under `~/.codex/automations` (`qr-*`, excluding `qr-weekend-*`) plus weekday Claude scheduled tasks whose project is `QuantRabbit` or `QuantRabbit-live`; explicit `QR_WEEKEND_CODEX_TASKS` / `QR_WEEKEND_CLAUDE_TASKS` env lists override discovery. `qr-self-improvement-watch` is pause-managed but never restore-enabled: Monday restore must force it to `PAUSED` even if an older Saturday snapshot recorded `ACTIVE`, so the repair agent cannot turn itself back on automatically. It also snapshots and stops DecaBot launchd labels `com.decabot.ai`, `com.decabot.monitor`, and `com.decabot.review`, then restores only labels that were loaded in the snapshot; `com.decabot.live` remains unmanaged unless the operator explicitly requests it. While `~/.codex/quant_rabbit_weekend_task_state.json` has `mode=paused`, `sync-live-runtime.sh` may accept `qr-trader` as `PAUSED`; without that guard state, a paused trader automation remains a sync blocker.
- **Every cycle reads the same `docs/SKILL_trader.md`.** Changes there affect every subsequent cycle whichever model runs.
- **Main trader cadence/model policy (2026-06-30).** Active Codex automation `qr-trader` runs `gpt-5.5` with `reasoning_effort=high` every 60 minutes. This is the deep discretionary trader cycle, not the risk-monitoring clock. Guardian probe/router monitoring remains deterministic, non-LLM, and frequent, with event-driven GPT-5.5 high/read-only wake review on state changes.
- **Hourly AI market-read authority is a bounded, content-addressed veto (2026-07-11).** The hourly `qr-trader` must not pass a deterministic `TRADE` receipt off as an AI decision. `trader-draft-decision --output data/trader_decision_baseline.json --market-read-evidence-packet data/market_read_evidence_packet.json` writes a deterministic baseline plus a SHA-256-bound evidence packet. The scheduled Codex `gpt-5.5` / high pass then writes only `data/codex_market_read_overlay.json`, reviews the latest genuinely resolved score-eligible market-read prediction, states what failed and what changed, gives a counterargument, and supplies numeric quote-relative 30m/2h target/invalidation geometry plus numeric forced-trade entry/TP/SL. `RANGE` requires lower/upper target rails around the current quote and strictly wider lower/upper invalidation rails. `trader-apply-market-read` must re-hash the baseline and every evidence source immediately before atomically publishing `data/codex_trader_decision_response.json`; stale baseline/evidence/model output, missing prior-error review, incomplete numeric geometry, unknown overlay fields, or a changed execution/risk envelope rejects the merge. The AI may `ACCEPT_BASELINE`, or may downgrade only a deterministic `TRADE` to `WAIT` / `REQUEST_EVIDENCE` with an explicit veto reason. It may never upgrade a non-trade baseline to `TRADE`, choose a different lane, change selected units/order ids/risk fields, widen permission, or authorize a broker write. A final `TRADE` therefore requires fresh `decision_provenance.author_kind=CODEX_MARKET_READ`, `model=gpt-5.5`, `reasoning_effort=high`, matching market-read and final-envelope digests, `execution_fields_preserved=true`, `risk_envelope_not_expanded=true`, and `live_permission_granted=false`; the deterministic verifier and all normal gateway/risk/ownership gates still decide execution. A valid AI veto may override only the generic campaign/attack pressure that otherwise forces the baseline trade; close-first, exposure, stale evidence, risk, ownership, pending cleanup, and other hard gates remain authoritative. `run-autotrade-live.sh` must never replace a Codex-authored receipt with its deterministic auto-draft when that receipt becomes stale, rejected, or consumed: the stale receipt blocks new risk while the ordinary gateway-maintenance cycle continues, and the next hourly AI pass must author a fresh overlay. QuantRabbit code must not call a model API or API key for this path; the scheduled Codex task is the named discretionary author.
- **Watchdog evidence binding exception (2026-07-11).** In the market-read packet, “re-hash every evidence source” means material-state hashing for `qr_trader_run_watchdog`. Contract `QR_TRADER_WATCHDOG_SAFETY_STATE_V1` binds health/severity/issues, cadence and automation/weekend state, current material guardian-receipt identities, and the no-write execution boundary. It deliberately excludes observation clocks, running age counters, log-query timestamps/excerpts, rendered review timestamps, and last-run timestamps that the current handoff itself may advance. - **Guardian market-state wake and tuning queue (2026-07-10; identity/lifecycle revision 2026-07-11).** The frequent guardian reads complete M1/M5/M15 candles for every open-position pair regardless of owner plus its bounded candidate frontier. Technical events record the latest complete M1/M5/M15 source-candle watermarks independently of the consensus timeframe surface. A quick/full chart-packet switch, a regenerated chart whose comparable fast watermarks did not advance, or a present-but-empty/invalid watermark contract cannot turn derived regime/family/structure drift into new market information; an independently material live-price displacement remains reviewable. Major-figure failed-acceptance watches must not be emitted while the quote already violates the spread-anomaly cap. The major figure itself is semantic identity; bid/ask changes are bounded observation evidence, not one new queue obligation per tick. Dispatcher events not selected in the current GPT pass, or deferred by the shared live lock/stale broker truth, remain in bounded `pending_dispatches` state and must be drained after the router overwrites its next transient escalation. Failed GPT attempts retry the exact failed observation under backoff/max-attempt/TTL limits; a newer raw quote with the same dedupe key cannot replace that pending observation or reset its budget, but it must be retained in that key's bounded successor list and promoted after the older obligation succeeds. Codex quota exhaustion is `CODEX_USAGE_LIMIT`, not `SCHEMA_INVALID`: it skips schema repair, preserves the exact pending event, and uses a long capacity backoff so a 30-second monitor cannot burn repeated quota attempts. An accepted material review writes `data/guardian_tuning_work_order.json` schema v2 queue revision 4. Same-semantic material price/lane changes append bounded `observations[]` evidence instead of either minting another obligation or disappearing. Distinct pending obligations remain in `work_orders[]`, while bounded `terminal_history[]` preserves consumed/superseded experiment IDs so a recurrence can carry `reopened_from_terminal` and does not repeat a failed experiment blindly. Dispatcher, review enrichment, experiment preparation, and terminal writers must all acquire the stable sibling lock `data/guardian_tuning_work_order.json.lock` and compare-check the source SHA immediately before atomic replace. The hourly AI must never edit the queue JSON directly. The current forward-only experiment, exact-lane precommit, fixed-zero evaluator, complete executed-trade baseline, stage→terminal→confirm activation, and crash reconciliation contract is defined authoritatively in §15; any older frozen/in-sample workflow is invalid. The bounded queue must fail closed with `WORK_ORDER_QUEUE_FULL` when all slots are occupied; it never inserts a new item by dropping an older pending obligation and immediately releases only the dedicated queue-full retry budget when a durable slot becomes available. Guardian GPT must attach a structured, boundary-preserving `bot_tuning_review` for every tuning reason, naming only the selected pair, one exact family-allowlisted non-risk parameter, and one falsifiable hypothesis/experiment. Missing, incomplete, cross-pair, execution-instruction, risk/lot/permission-changing, unknown-parameter, or otherwise unsafe review content remains `STRUCTURED_REVIEW_REQUIRED`: the work order is preserved for acquisition, but the dispatcher neither publishes/supersedes an action receipt nor acknowledges, starts the gateway/action cycle, or executes it. Only after a `TEST_REQUIRED` review and its complete evidence may the lifecycle tool atomically mark the item `CONSUMED` or `SUPERSEDED`. A work-order read/write/concurrent-update/full-queue/structured-review/experiment/evidence-validation failure is not an accepted dispatcher acknowledgement, and a later recurrence of a terminal semantic state reopens it. ## 3. Prime Directive

- Build from broker truth outward.
- Enforce broker gateway and risk contracts in code before prompt or strategy-prose optimization.
- Express discretionary autonomy as executable receipts, not vague market prose.
- Default to dry-run / read-only behavior unless live gates are explicitly satisfied.
- **Adapt to live market conditions, not to fixed numbers.** Sizing, stop distance, target distance, spread tolerance, exposure budget, and pacing must all be derived from current ATR, regime, spread, equity, daily progress, and liquidity — not from JPY / pip / multiplier literals baked into code or prompts.

## 3.1. Critical Thinking / Anti-Absolutism

Every QuantRabbit guard, profile status, prompt rule, score penalty, audit blocker, and safety wrapper is a falsifiable hypothesis, not absolute truth. A rule created to repair one failure can become the next profit blocker. Agents must therefore audit self-created mechanisms with the same skepticism applied to market forecasts.

- Treat every blocker as guilty until it proves current value: when profitability is poor or opportunity coverage is thin, rank `live_blocker_codes`, cancel regrets, missed MFE, stale artifacts, and exit-reason drag by actual opportunity cost before adding another rule.
- A guard must publish its escape condition in executable form. If prose says "require a new vehicle or market-structure proof" but code never recognizes that proof, the guard is a bug.
- Prefer downgrade paths over permanent bans: use `WARN`, smaller size, non-market triggers, shorter TTL, attached TP, or explicit evidence collection when broker truth, current forecast, rail geometry, and gateway risk checks still support a bounded attempt.
- Negative history is advisory unless it is current, well-sampled, and still matches the present vehicle. Do not let stale or over-segmented strategy memory block a fresh market-proof `LIMIT` / `STOP-ENTRY` that has current forecast support, correct structure, bounded risk, and attached TP.
- Before preserving a blocker, state the strongest counterargument against it. If the counterargument is supported by live ledger, execution-timing audit, or capture economics, change the blocker or make it evidence-conditional.
- New guards need two tests: one proving the original failure remains blocked, and one proving the guard does not block the valid current shape it claims to allow.

## 3.5. No Thoughtless Hardcodes or Fallbacks

This rule applies to every file the trader reads or writes — `src/quant_rabbit/**`, `tools/**`, CLI flags, scheduled-task SKILLs, prompts, and config.

- **Hardcoded JPY / pip / multiplier values are forbidden** unless they are documented, named (`InstrumentSpec.normal_spread_pips`, `RiskPolicy.daily_risk_pct`, etc.), and clearly marked as a *test default* or a *broker-spec constant* — not as a production risk decision.
- **Silent fallbacks to a literal are forbidden.** If a required input is missing, raise (or emit a `*_MISSING` issue with severity `BLOCK`) so the operator sees it. Falling through to `RiskPolicy().max_loss_jpy` or `8 pips` or `1.5 R` quietly is the failure mode this contract was written to prevent.
- **Risk caps must be equity-derived, and per-trade ≠ per-day.** The daily-target ledger (`campaign-day opening balance × daily_risk_pct`) is the **whole-day** budget; keeping that denominator fixed prevents a realized loss from resetting the day's allowance. Gross current-day trader-attributed realized loss is non-refillable: `realized_loss_spent_jpy = Σ max(0, -(realized_pl_jpy + financing_jpy))` over each `TRADE_CLOSED` / `TRADE_REDUCED` broker event whose `trade_id` joins to a QuantRabbit `ORDER_FILLED` entry with its own lane or a matching gateway/accepted lane. Manual/tagless trades are excluded, and a later profitable close or partial reduction must not erase an earlier losing event. Persist `daily_loss_capacity_before_open_jpy = max(0, daily_risk_budget_jpy - realized_loss_spent_jpy)` and `remaining_risk_budget_jpy = max(0, daily_loss_capacity_before_open_jpy - fresh_open_risk_jpy)`. The gateway portfolio cap is **capacity before open risk**, never `remaining_risk_budget_jpy`: execution re-reads the freshest state immediately before validation, takes the tighter of any earlier explicit cap and current capacity, then subtracts fresh broker open risk, pending risk, and candidate/batch risk exactly once. Passing the already-open-subtracted remainder into that gate is a double-counting bug. Every broker-mutating entry POST (direct, each batch candidate, AutoTradeCycle, and guardian handoff) must therefore perform this final sequence inside `LiveOrderGateway`: sync the execution ledger, fetch a fresh broker snapshot, require exact ledger/snapshot `lastTransactionID` equality, refresh `DailyTargetLedger` from that ledger and the same snapshot, resolve the final per-trade/capacity values, then account open + pending + candidate risk once against that same snapshot. Missing sync capability/path, a baseline-only ledger, transaction mismatch, target-refresh failure, or a legacy state that cannot publish the required fields is a hard BLOCK. Intent metadata risk caps may tighten the fresh/policy cap but must never widen it. At this pre-POST boundary, only fresh target status `PURSUE_TARGET` permits new entry risk; `TARGET_REACHED_PROTECT`, repair/exhausted states, and missing or unknown statuses block. Ordinary final downsizing rebuilds and re-lints the broker order; a verified predictive SCOUT whose units or geometry would change is blocked for regeneration and re-verification rather than silently mutated. The base **per-trade** cap is `daily_risk_budget_jpy / target_trades_per_day`, while the automated minimum meaningful shot is recalculated from the latest broker raw NAV (`current NAV × RiskPolicy.min_per_trade_risk_pct`) on every broker refresh; the final per-trade cap may only tighten to remaining loss capacity. The state must persist `sizing_nav_jpy`, `base_per_trade_risk_budget_jpy`, and `per_trade_risk_pct_nav` so a JPY cap cannot masquerade as a current-equity percentage. This final per-trade result is what flows into `intent.metadata['max_loss_jpy']`; SL distance and quote-to-JPY conversion then determine units, while margin/portfolio gates may only reduce or block them. The split is mandatory: a single losing trade must not exhaust the day's budget, and a falling or rising NAV must immediately shrink or compound the next sizing decision. `target_trades_per_day` must come from `ai_test_bot_backtest.firepower.required_trades_per_day_at_observed_expectancy` when available; `RiskPolicy.target_trades_per_day = 10` is only the no-evidence fallback. Override via `--target-trades-per-day` on `daily-target-state` for an explicit operator pace. Never derive either cap from a JPY literal stored in code or in a stale state file.
- **Trader score sizing is trim-only; proof contracts own every increase.** `TraderBrain` may use current lane score to reduce a generated size, but `size_multiple_max` is hard-capped at `1.0` and missing settings also default to `1.0`. A score is not realized expectancy proof and cannot widen `effective_max_loss_jpy`. Larger risk than the currently loss-asymmetry-capped shot may come only from the existing pre-unit exact-TP, predictive-SCOUT, or macro confidence contracts, and the final result remains bounded by fresh daily per-trade capacity, gross-loss non-refill capacity, current broker NAV × 1%, margin, portfolio, pair, and concurrent exposure gates.
- **Macro-event confidence changes the fraction, never the fresh absolute cap.** A same-direction factual post-release `event_surprise_followthrough` may use a larger tranche as its calibrated signal confidence rises, but it must remain inside the smaller of the current daily-target per-trade cap, current broker NAV × 1%, and gross-loss non-refill capacity. Confidence below the 0.80 event threshold uses a SUB_THRESHOLD 0.10 evidence tranche rather than falling back to a full routine-size trade; eligible confidence maps to LOW/MEDIUM/HIGH risk fractions 0.25/0.50/1.00. The resulting cap and current SL loss per unit determine integer units. The old `base cap × 3` expansion is forbidden because a generator packet at 1,500 JPY could never pass a fresh 500 JPY gateway policy. Intent metadata records the band/fraction for audit only; final pre-POST broker truth may tighten it and stale metadata may never widen it. This confidence rule grants no extra concurrent-risk permission: ordinary portfolio capacity still applies, and predictive SCOUT remains separately capped at 1% per trade / 2% concurrent by its atomic reservation contract.
- **Ordinary GPT entry receipts are one-shot broker signals.** Immediately before each ordinary live entry POST, after ledger/target reconciliation and every final BLOCK check, `LiveOrderGateway` must commit a SQLite `BEGIN IMMEDIATE` claim keyed by the canonical accepted GPT decision, `parent_lane_id`, and `forecast_cycle_id`. The same receipt cannot be reused for that parent even if intent metadata is edited, and the same parent/cycle cannot be reclaimed by writing a cosmetically new receipt. A later `ADD` therefore requires both a new accepted receipt and an independent forecast/trigger cycle. The reservation is committed before network I/O and is never released: success records the broker response, an ordinary client exception records `POST_OUTCOME_UNKNOWN`, and a process crash leaves `RESERVED_PRE_POST`; all three consume that signal so an ambiguous outcome fails closed. Predictive SCOUT keeps its separate vehicle/signal reservation contract. The same final broker snapshot must also rebuild trader pending geometry, parent-lane keys, and position-plus-pending occupancy; any duplicate or newly exhausted occupancy that appeared after initial validation blocks the POST.
- **Campaign-day opening balance is immutable after audit.** On a new campaign day, reconstruct the opening from current broker cash balance minus the complete account-wide current-day realized P/L plus financing delta (manual/tagless included) and minus recorded net capital flows. Every close/reduction must retain valid raw `ORDER_FILL` JSON whose matching trade component reconciles to normalized realized P/L and financing; normalized values without that cash-field audit cannot authorize risk. Daily `OANDA_TRANSACTION` financing is account-wide for reconstruction; its raw `positionFinancings[].openTradeFinancings[]` components join back to system entries for trader progress and gross loss spend, while manual components remain excluded from trader attribution. Close-event financing and daily-financing transactions are distinct broker cash events and must each be counted exactly once. Broker `TRANSFER_FUNDS.amount` must reconcile to the same-day capital-flow sidecar before that sidecar can be subtracted; a missing/mismatched flow or any unsupported nonzero cash component must fail closed. Once audited, the opening must not be re-anchored later that day by system or manual profit/loss. An incomplete/stale account delta, invalid/unreconciled financing JSON, or a mismatch against the persisted same-day opening emits a visible campaign-opening audit blocker and authorizes zero fresh-entry loss capacity. A provisional first-run/new-day target may remain visible for compatibility and may be replaced once after ledger synchronization; it never grants trading capacity.
- **Daily pace visibility is advisory and basis-separated.** `daily_target_state.json`, its report/CLI output, and the GPT daily-target packet must preserve the top-level `ai_test_bot_backtest.firepower` stretch evidence as `uncapped_required_trades_per_day`, `uncapped_required_trades_per_day_basis_return_pct`, `observed_trades_per_day`, `observed_expectancy_jpy_per_trade`, and `frequency_multiple_required`, even when an explicit or carried CLI pace wins. These fields diagnose expectancy and turnover shortfall only: they must not create a `BLOCK`/`WAIT`, grant live permission, raise lot size, alter `target_trades_per_day`, or change per-trade/daily/portfolio risk math.
- **Geometry must be ATR-derived.** SL distance and TP distance must scale with current ATR (per pair, per timeframe, per regime). `spread × N` is a *floor*, not the geometry. If pair-charts data is unavailable, the cycle waits — it does not invent numbers.
- **Reward / risk targets must be regime-derived.** A single fixed `target_reward_risk` for all conditions is the same anti-pattern as a fixed loss cap. Trend regimes deserve wider targets; range regimes deserve faster rotation.
- **Spread tolerance must be liquidity-derived.** A static `2.5 ×` multiplier across all sessions ignores Tokyo open vs London close. Tighten in liquid hours, loosen in thin hours, and BLOCK above session-aware caps.
- **No "default-to-yesterday" stale persistence.** State files (e.g. `data/daily_target_state.json`) must invalidate on a new campaign day or when the snapshot post-dates the file by more than one cycle. Reading a stale value and pretending it is current is a silent fallback in disguise.
- **Every numeric constant in production code requires a docstring or comment** stating: (a) what market reality it represents, (b) why it is constant rather than market-derived, and (c) what should replace it if it ever needs to be changed. If you cannot write that comment, the value should not be a constant.
- **When in doubt, fail loud.** A `BLOCK` issue with a clear message is always preferable to a guess. The operator can act on a blocker; the operator cannot recover from a silent miscalculation.
- **Broker integer-unit floor (`MIN_PRODUCTION_LOT_UNITS = 1`, 2026-07-10; supersedes the 2026-05-12 1000u floor).** OANDA broker truth on this account already accepted 201u, 322u, and 2u orders. Lot size multiplies both gross P/L and spread cost linearly, so sub-1,000u size alone does not make a positive-pip vehicle structurally negative; economic validity is determined by bid/ask spread, executable TP/SL geometry, slippage/cost evidence, and post-cost expectancy. `_risk_budgeted_units` must therefore preserve `current NAV × dynamic risk% ÷ SL loss per unit` at one-integer-unit precision, then cap it by margin and portfolio capacity. A recovery hedge must wait for a pullback, retest, or non-market trigger instead of buying/selling the already extended price. `RiskEngine` uses the range/recovery reward floor (`range_min_reward_risk`) rather than the fresh trend-entry floor. - **Hedge timing classes and unwind discipline (2026-05-20).** A same-pair hedge intent must publish `metadata['hedge_timing_class']`: `LOCK_GAIN` for profitable existing exposure, `REVERSAL` for underwater exposure with machine-readable reversal confirmation, or `CONTINUATION` for underwater exposure still moving against the original side without reversal proof. ## 6. Autonomous Pro Trader Requirements

- The operator is the discretionary decision layer, but every decision must be grounded in structured broker state, mined history, current market context, and risk geometry.

- **WAIT discipline.** A WAIT decision is only valid when it cites a specific gate that fired, by name from this contract or from `data/`. Generic prose ("Golden Week thin liquidity", "EVENT_RISK", "UNCLEAR regime") is not sufficient. A precedent recorded at per-trade=4000 has 1/10 the weight under per-trade=400. Cite the rescaled figure or do not cite the precedent.

## 10. Position Protection

- Missing SL repair remains a **repair requirement** when SL mode is enabled; in SL-free mode it is intentionally suppressed. Missing broker TP is **not** auto-created by default: a manually deleted TP or runner created without `takeProfitOnFill` is treated as a no-broker-TP runner unless `QR_ENABLE_MISSING_TP_REPAIR=1` is explicitly set. Exception: if the position is already in profit and the latest forecast / technical stack no longer justifies carrying the runner into the next session, `tp-rebalance` may add an **insurance TP** near current reward-side price. That is MFE-capture insurance, not a loss-side stop and not permission to cap every runner.
- TP rebalance applies only to existing broker TP orders on trader-owned positions and operator-managed manual/tagless positions whose operator packet does not set `auto_tp_modify_allowed=false`. SL repair / tightening applies only to trader-owned positions. Operator-managed manual/tagless positions are never stop-lossed or market-closed by the trader.
- Profitable protected positions can tighten SL to break-even or better.
- **SL-free profit-only break-even/profit-lock stop (2026-05-20; TP-progress gate added 2026-06-03).** `QR_TRADER_DISABLE_SL_REPAIR=1` still means missing SL is intentional and must not be "repaired" on underwater or noise-bound positions. Exception: a trader-owned SL-free position that is already profitable may receive `BREAK_EVEN_STOP` at the original entry price or better only after executable profit pips clear both (i) a fresh market-derived micro-noise envelope: current spread plus fresh M5 ATR and quick M1/M5 realized range from recent candles inside the current M5 management window, and (ii) the TP-progress gate (`QR_PROFIT_BREAK_EVEN_MIN_TP_PROGRESS`, default 60% of current broker TP distance) when a broker TP exists. Stale pair-chart volatility must not be used; if fresh quick volatility is unavailable, the stop is deferred rather than falling back to spread alone. When profit exceeds those gates, the stop is placed behind the current executable price by the same envelope and clamped so it can never be worse than entry. This is not initial SL repair, not trailing SL, and not permission to place loss-side stops; it is a profit-to-flat/profit-lock escape hatch once MFE is large enough that a reversal should not be allowed to turn the trade red, without turning a planned TP trade into a micro-scalp. Manual/tagless positions remain TP-only and must never receive this stop.
- **Bollinger rail continuation handling (2026-05-20; forecast-aware adaptive TP gate added 2026-06-03).** When an already-profitable trader-owned SL-free position has a same-direction BB rail / oscillator setup — e.g. SHORT while M1/M5 price is at the upper Bollinger rail with StochRSI/Williams/MFI overbought, or LONG at the lower rail with oversold evidence — `PositionManager` must not shrink a valid opposite-rail TP just because the micro matrix says HARVEST. If the existing TP already sits at the opposite M5 BB rail, keep that TP and use only the profit-side BE/profit-lock stop. Outside the BB-rail exception, a HARVEST/NARROW contraction of an existing reachable broker TP must prove current market-read MFE risk before replacing the TP: the latest pair forecast must no longer support runner carry and at least two independent technical pressure signals must show exhaustion / fractured agreement / rail risk. Only after that forecast+technical read is present may the adaptive TP-progress gate (`QR_ADAPTIVE_TP_CONTRACTION_MIN_PROGRESS`, defaulting to the same 60% majority-of-target boundary as profit-lock) approve the contraction. A strong same-direction forecast keeps a reachable TP running even when simple progress has cleared. The only early-contraction exception is a stale-wide TP already beyond `QR_TP_REBALANCE_MAX_DISTANCE_ATR × operating ATR`; it may contract when current profit is still smaller than operating ATR, or when the same forecast+technical MFE-risk stack is present. This prevents a +1 pip micro-scalp from replacing a still-reachable +7 pip TP while preserving the stale-target repair that saved trades whose original TP was never realistic. This captures the operator's "BB upper, likely down" read: the rail is continuation evidence for the current SHORT, not a reason to close the short or pull the target prematurely.
- Contradicted trader-owned positions can be closed.
- **Existing SL cannot be widened.**
  - **Existing TP is dynamically rebalanced every cycle** (2026-05-13 update; manual/tagless included 2026-05-15; missing-TP creation disabled by default 2026-05-15; insurance-TP exception 2026-05-15; forecast-harvest contraction added 2026-05-20; stale profitable TP contraction added 2026-05-21; router/verifier enforcement added 2026-05-20; explicit operator-manual TP opt-out enforced 2026-07-06). The `tp-rebalance` CLI command (invoked from the trader skeleton between intent pricing and gateway execution) recomputes TP per open trader-owned or operator-managed manual/tagless position using `_market_derived_reward_risk(chart_context) × current_ATR` when a broker TP already exists, unless the operator-manual packet explicitly sets `auto_tp_modify_allowed=false`. `trader-prompt-route` must also run the same calculation in dry-run memory and route to `position_management` whenever an adjustment exists, and `gpt-trader-decision` must reject WAIT / REQUEST_EVIDENCE with `TP_REBALANCE_REQUIRED` while that sidecar is executable. It creates a missing TP only when `QR_ENABLE_MISSING_TP_REPAIR=1` explicitly opts into repair, or when an already-profitable TP-less runner needs insurance because forecast confidence / horizon, next-session timing, price percentile, MTF agreement, 24h range exhaustion, RSI/StochRSI/Williams, Bollinger, or Donchian evidence says the maximum favorable excursion is at risk. A profitable position that already has a broker TP may contract that TP when the latest forecast no longer supports directional runner carry and the technical stack also shows MFE risk; the replacement must be a usable structural HARVEST anchor from `price_action.structural_tp_target`, skipping anchors inside the market safety margin rather than falling back to a guessed pip value. Separate stale-harvest repair is allowed when an attached broker TP is more than `QR_TP_REBALANCE_MAX_DISTANCE_ATR × operating_ATR(M5/M15)` away and the current profit is still smaller than operating ATR, or when a wider technical exhaustion stack is already present; in that case the replacement is a near-market reward-side TP, not a loss-side stop. Once PositionManager or tp-rebalance has set a reachable HARVEST TP inside the operating-ATR cap, later TP passes must not expand it back into a runner merely because unrealized P/L changes before the broker fills that harvest. When the new desired distance differs from the existing TP by ≥ `QR_TP_REBALANCE_HYSTERESIS_PIPS` (default 10), the broker's TP order is replaced. Invariants enforced by `strategy/tp_rebalancer.py`: (a) TP must stay on the correct side of entry (LONG above, SHORT below) so the rebalance cannot lock in a loss, (b) TP must stay ≥ `QR_TP_REBALANCE_MIN_TP_TO_MARKET` pips from current price so it cannot fire on the same tick, (c) TP distance is capped at `QR_TP_REBALANCE_MAX_DISTANCE_ATR × ATR` to bound greed, (d) SL is NEVER touched — SL-free positions remain SL-free, including manual/tagless positions. External positions are skipped. Kill switches: `QR_DISABLE_TP_REBALANCE=1` for the whole pass, `QR_DISABLE_INSURANCE_TP=1` for only missing-TP insurance. User directives 2026-05-13/15/21:「エントリー時だけではなく、途中で伸び縮みできるようにしてほしい。市況によって。それって当たり前だよね？」「基本的には利を伸ばす方向がいいよ。ただ、伸ばすことに固執して、最大含み益の時点で決済できないと意味ない。だからテクニカルをたくさんみなきゃいけない。」「今のショートポジ、逃げておくべきじゃない？プラスのうちに。そういう判断できてる？TP長いんじゃない？」. - **New-entry TP execution has two explicit modes** (2026-05-15; stale failed-break initial TP repair added 2026-05-21; technical-HARVEST RR floor clarified 2026-06-17). `intent_generator` always computes a virtual TP for risk/reward validation, but `intent.metadata.attach_take_profit_on_fill` decides whether `LiveOrderGateway` attaches broker `takeProfitOnFill`. `ATTACHED_TECHNICAL_TP` is used for range rotation, failed-break setups, small-wave / low-ADX tape, low TF agreement, exhausted 24h range, or thin liquidity; the TP is anchored on the nearest structural HARVEST target from `price_action.structural_tp_target` when available. If a failed-break attached HARVEST target has no usable structural anchor and the ATR/RR fallback would be more than `QR_HARVEST_TP_MAX_OPERATING_ATR_MULT × operating_ATR(M5)` away, the broker TP must use the minimum acceptable reward-side operating target that clears the technical-HARVEST RR floor (`RiskPolicy.technical_harvest_min_reward_risk`, default 1.0) and spread floor instead of inheriting a distant SL-free virtual risk distance. `RiskEngine` applies that 1.0R floor only to `BREAKOUT_FAILURE` receipts whose metadata declares `opportunity_mode=HARVEST`, `tp_target_intent=HARVEST`, and `tp_execution_mode=ATTACHED_TECHNICAL_TP`; `TREND_CONTINUATION`, forecast runner, spread, loss-cap, forecast/profile, stale-quote, and gateway checks do not get relaxed. - **S5 forecast-failure SCOUT is forward-evidence only and current-NAV risk-sized** (2026-07-10; variable sizing added after the operator required asset-percentage decisions). A canonical `LIVE_GRADE_DAILY_STABLE` contrarian S5 bid/ask rule may seed a `failure_trader` `BREAKOUT_FAILURE` experiment only when the current pair/side/forecast direction/confidence/horizon bucket matches the packaged rule and its full rule-authentication digest. Packaging timestamps, report paths, and rule-file provenance do not change that digest. Failure memory uses a stable vehicle id made from the market selector plus executable pair/side/method/order/TP/SL shape; **units and NAV are deliberately excluded**, so changing account size cannot erase cooldown or quarantine. Units are instead finalized before GPT verification from `latest broker NAV × exact-vehicle forward risk tier ÷ canonical SL loss per unit`, floored to the nearest positive integer broker unit and capped again by the ordinary per-trade, daily, portfolio, margin, spread, news, guardian, and gateway gates. Only a result below one unit is blocked; 1–999u orders remain valid when every cost and risk gate passes, and their smaller contribution is handled by higher independent-signal frequency rather than upward rounding. `config/predictive_scout_policy.json` schema v2 defines exact-vehicle tiers: `DISCOVERY` 0.10% NAV (0–4 normalized exits), `EMERGING` 0.25% (≥5, net R > 0 and PF(R) ≥ 1.0), `ESTABLISHED` 0.50% (≥10, PF(R) ≥ 1.2 and positive-day rate(R) ≥ 2/3), `STRONG` 0.75% (≥20, five active days and one-sided 95% mean-R lower bound ≥ 0), and `PROVEN` 1.00% (≥30, lower bound > 0, PF(R) ≥ 1.2, all fills resolved, complete signal attribution, no duplicate signal). Historical replay and generic TraderBrain scores may qualify or rank a hypothesis but can never raise its tier; only normalized forward outcomes from the same stable vehicle can do so. The historical replay entered immediately after forecast emission and is therefore **not** proof of the generated passive LIMIT retest; the intent and gateway must label it `UNPROVEN_PASSIVE_LIMIT`, `predictive_scout_rule_is_vehicle_proof=false`, and `predictive_scout_promotion_allowed=false`. A fresh broker NAV tick may leave the verified units unchanged only while planned SL risk remains inside both the original and current tier cap; otherwise the gateway blocks and the next cycle must regenerate smaller units. Any manual/operator-manual/tagless exposure on the same pair blocks both directions. Before every POST, one `BEGIN IMMEDIATE` transaction claims unique `(vehicle_id, forecast_cycle_id)` and enforces daily/concurrent reservations; a second POST for the signal is forbidden, broker `lastTransactionID` must match ledger sync state, and the sent receipt is indexed before return with vehicle/signal ids in broker extensions. Once filled, entry-time TP/SL/units are the experiment and no adaptive position path may modify or auto-close it. The scheduled digest and operator-review report may surface `PROOF_ELIGIBLE_FOR_OPERATOR_REVIEW`; neither auto-promotes nor guarantees future profit.

The baseline fixes the one exact `close_trade_ids` value and carries no entry lane or pending-order cancellation. The GPT market-read overlay may only accept this non-TRADE baseline and cannot create, remove, replace, or expand its close target. Structural authority must come from the separately verified timestamped H4 or structural position-management / position-thesis path. If the sidecar says the evidence is adverse-entry-buffer-only, missing-thesis, entry-thesis-lacks-invalidation, or time/forecast decay only, it stays soft and needs explicit Gate B. Softer close evidence still requires explicit operator authorization from one of two operator-controlled channels before a loss-side `CLOSE`: (i) `QR_OPERATOR_CLOSE_OVERRIDE=1` exported in the operator shell, or (ii) a fresh `data/.operator_close_token` file whose mtime is within `OPERATOR_CLOSE_TOKEN_FRESH_SECONDS` (5 minutes; see `gpt_trader.py`). ## 13. Engineering Conventions

### Package manager
- None. Use system Python with `PYTHONPATH=src`.
- Test with `PYTHONPATH=src python3 -m unittest discover -s tests -v`.

### Commit attribution
AI commits MUST include the line for the operator who authored the change:

- Codex commits:
  ```
  Co-Authored-By: Codex <noreply@openai.com>
  ```
- Claude commits:
  ```
  Co-Authored-By: Claude <noreply@anthropic.com>
  ```

If both contributed in the same commit, include both lines.

### Always keep commits current
- The working tree must be **committed before ending a session**, before switching operator (Codex ↔ Claude), and before any scheduled-task cycle that may run live (`autotrade-cycle --send`).
- Commit logical units as they finish — do not let unrelated changes pile up across sessions. Untracked or uncommitted work invalidates the audit trail and risks the next operator inheriting an unknown state.
- A committed change is not finished until it is pushed to the remote tracking branch. When guarded promotion moves `main` or `codex/live-trader-runtime`, push those branch heads too unless the operator explicitly says to keep the work local.
- Receipts and reports written into `data/` and `docs/*_report.md` are runtime artifacts; commit them only when they are part of a deliberate snapshot, not on every cycle (they are usually `.gitignore`-d under `data/`).
- Before starting work, run `git status` / `git log -1` to confirm the tree is clean and the previous operator's commit is the latest. If it is not, either commit the leftover changes (with the correct operator attribution) or ask the user before discarding.
- Never `--amend` a published commit, never force-push, and never skip hooks (`--no-verify`). Stack a new commit instead.

### Acceptance bar
- Add a **failing regression test** for known legacy failure modes.
- Add a **passing test** for new behavior.
- Document the live-risk failure mode for accepted rebuild components.
- Preserve dry-run receipt paths for new execution features.
- **No live side effect unless explicitly enabled.**
- Never promise market-guaranteed returns. Encode the +5 % pace marker as a review/protection obligation, audit ledger, incident trigger when a valid edge was missed, and bounded, testable execution behavior.

## 15. Guardian Tuning Experiment Provenance

- These rules supersede earlier guardian prose that described an in-sample/frozen-evaluator threshold run. A valid `TEST_REQUIRED` review must first fix the candidate and one exact five-part `lane_id` in `desk:pair:side:method:vehicle` form. Only entries opened strictly after the queue's immutable `structured_review_completed_at_utc` may enter its cohort; historical closes are hypothesis-generation evidence, never accepted proof.
- Review acquisition is monotonic per current observation. `tools/guardian_tuning_review_enrich.py` may bind either a boundary-valid `NO_CHANGE_INSUFFICIENT_EVIDENCE` review with no adjustment or a boundary-valid `TEST_REQUIRED` review. Every no-change review must carry a structured allowlisted acquisition kind, bounded project-relative evidence source, integer new-sample target, and exact non-executing success condition; vague wait/monitor prose is invalid. Repeating identical content for the same current observation is idempotent. For that same observation, a pending `NO_CHANGE_INSUFFICIENT_EVIDENCE` review may upgrade to `TEST_REQUIRED` when the exact lane, active value, candidate, and forward experiment become available, while `TEST_REQUIRED` cannot downgrade. A materially new latest observation invalidates the old binding and must receive its own review; it may be `NO_CHANGE_INSUFFICIENT_EVIDENCE` even when the prior observation was `TEST_REQUIRED`. Neither review is terminal, grants live permission, or frees queue capacity. `trader-support-bot` must build read-only `guardian_tuning_acquisition` progress from the review boundary, canonical system entries/outcomes, and strict entry-thesis JSONL. It freezes the first requested N exact-lane entries, or pair+family entries when the review has no exact lane, before checking outcomes; a later resolved entry cannot replace an earlier unresolved one. It reports entry/pre-entry/resolution/complete counts plus `WAITING_FOR_ENTRIES`, `WAITING_FOR_RESOLUTION`, `COLLECTING_WITH_SIGNAL_DEFECT`, `ACQUISITION_SOURCE_DEFECT`, or `READY_FOR_GPT_REVIEW_UPGRADE`. Only the last status asks GPT to reconsider a monotonic upgrade; it does not perform one automatically. Replay/backtest may generate a hypothesis but never advances this forward-acquisition count or proves the candidate. Support reporting treats unsupported revision, counter mismatch, missing identity, or safety-boundary failure as queue-integrity P1 with unknown counts, never a zero backlog.
- The same review helper accepts a bounded `--manifest-json` for the hourly multi-item obligation. A batch item contains only exact `work_order_id`, exact current `expected_observation_id`, and the structured `review`. The helper must size/depth/string-check the manifest before parsing it as trusted work, acquire the stable queue lock once, reread and revalidate every item plus monotonicity against those current bytes, update only in memory, and perform one whole-queue compare-checked atomic write. Any malformed item, stale observation, conflict, concurrent writer, or write failure leaves the queue bytes unchanged; a partial batch success is invalid. Single-item mode remains available for one later observation or monotonic upgrade.
- Version 1 supports only `forecast_confidence_floor`. It is reconstructed from the exact `forecast_history.jsonl` row observed strictly before entry and joined through `entry_thesis_ledger.jsonl`; score/lookback/weight/execution parameters remain unsupported until they have equivalent append-only pre-entry truth. The reviewed `current_value` must equal the active pair/lane runtime setting and `candidate_value` must be a tightening inside `0..1`.
- Every new gateway-recorded entry thesis used by guardian tuning must bind `context_evidence.order_id`, the canonical exact five-part `context_evidence.lane_id`, `context_evidence.forecast_timestamp_utc`, and `context_evidence.forecast_cycle_id`; its thesis timestamp must be the broker `ORDER_FILL.time`, not the local recorder clock. Immediate fills derive those values from the OANDA fill response, while pending entries persist the create-order identity and preserve the same links when the later fill promotes the pending thesis. New rows may also carry `context_evidence.guardian_tuning_signal_state`, a bounded allowlist copied only from immutable pre-entry intent metadata (family/regime/structure/ATR and failed-acceptance context). `FAILED_ACCEPTANCE` is valid only from the explicit boolean M5 failed-break predicate matching the intent side; a `BREAKOUT_FAILURE` method label alone is not a predicate. The intent must preserve that predicate and its reclaimed acceptance boundary so the acquisition reader can reject missing or opposite-side evidence. Fill results other than immutable identity/time, realized P/L, exit/outcome fields, unknown keys, non-finite numbers, and oversized values are excluded. These links satisfy an `ADD_PREENTRY_SIGNAL_LOG` acquisition step for later evaluator work; by themselves they neither make a non-v1 parameter evaluable nor grant tuning, risk, or trade permission.
- Build source data only with `tools/guardian_tuning_cohort_builder.py --work-order-id ... --expected-observation-id ... --lane-id ...`; `--lane-id` must equal the exact five-part lane fixed in the review. At preparation, the recorded SQLite ledger and append-only thesis/forecast source tips must equal the current canonical source tips. The builder snapshots one SQLite ledger state and freezes the first 20 canonical attributed entries for that lane opened strictly after review, using the same gateway-to-fill attribution as canonical economics. Fewer than 20 entries, or any unresolved entry among those first 20, keeps the experiment pending; later resolved entries cannot replace an earlier unresolved entry. It reconciles entry units and financing-adjusted canonical outcomes, verifies the exact pre-entry forecast, derives the cutoff/watermarks itself, and normalizes each result to JPY per 1,000 units. After the cohort is frozen, later source appends are allowed only when the frozen prefixes still revalidate. Hand-authored signals, outcomes, subsets, cutoffs, review timestamps, and post-outcome lane selection are invalid.
- `tools/guardian_tuning_evidence_builder.py init-run` binds that content-addressed 20-entry forward cohort, exact observation, current-approved evaluator hash/version, fixed metric/objective, and fixed zero threshold with a strict `improvement > 0` comparison. The baseline is all 20 trades that actually executed in the frozen forward cohort, including audited-support exceptions below the nominal current floor; only the candidate is filtered by its proposed hard floor. `run` recomputes with the trusted declarative evaluator imported from `src`; Python copied below `data/` is provenance only and is never imported or executed. Acceptance also requires positive candidate expectancy and at least 80% retention against that complete executed baseline.
- The lifecycle writer revalidates the canonical ledger/log prefixes and complete evidence chain on every read. `ACCEPTED_IMPROVEMENT` first captures an immutable activation-ledger prefix and fresh activation instant, stages a dormant override, atomically writes the revision-4 `CONSUMED` terminal and lifecycle head with that same anchor, then confirms only the matching work-order/experiment/evidence record before activation. Run `tools/guardian_tuning_override_reconcile.py` before ordinary routing to repair a crash between terminal commit and confirmation. Pending confirmation or malformed override state blocks live send; it never silently falls back to a looser floor. An activated override tightens only the proved exact five-part lane floor in `data/guardian_tuning_overrides.json`; all existing gates remain. Rejected or superseded evidence changes no runtime setting. Deep override attestation may be reused only inside one explicit gateway validation phase; the cache is context-local, never process-persistent, and must be cleared at the start of final pre-POST reconciliation so queue, manifest, monitor evidence, and ledger truth are reread immediately before a broker write.
- After reconciliation and before ordinary routing, run `tools/guardian_tuning_post_activation_monitor.py`. For each activated exact lane it freezes the first 20 canonical attributed entries whose ledger row and raw OANDA broker-entry time are both strictly after the immutable activation boundary, waits until those same 20 resolve, and revalidates raw broker identities/times plus the immutable SQLite prefix before recording one content-addressed normalized metric. Every later evidence read must also rebuild from the current full ledger and prove that the current cohort is still complete with the identical first trade IDs, samples, and metric; unrelated later appends are allowed, but a late earlier entry, newly unresolved outcome, or financing correction makes the sealed evidence stale and requires a new seal before commitment. The runtime resolver permits that exact lane only while fewer than 20 qualifying entries exist; as soon as the twentieth exists it returns `OVERRIDE_POST_ACTIVATION_MONITOR_PENDING`, so `RiskEngine` blocks every later live send even if the scheduled monitor was skipped. A positive metric commits `KEEP`; a non-positive metric atomically commits `QUARANTINE`, and `RiskEngine` blocks that exact lane. A successor tightening for the same lane may be staged only after the prior commitment is `MONITORED_KEEP_COMMITTED`, and the prior activation manifest, terminal evidence, monitor evidence, and current ledger truth must revalidate before stage, terminal commit, and confirmation; an unmonitored, quarantined, or stale-provenance override cannot be replaced to reset the sample boundary. A queue/state split during this monitor transition fails closed and is idempotently confirmed on retry, and an already-valid content-addressed monitor artifact is reused rather than minting retry orphans. The recorded previous/rollback value remains audit evidence only: this monitor never silently relaxes to it, and an executable rollback requires a separate proof and lifecycle contract.
- Completed, aborted, and stale semantic digests plus every used experiment-id digest enter separate monotonic bounded registries that fail closed when full and survive bounded terminal-history trimming. Prose, whitespace, metadata, labels, or reuse of an old experiment id cannot repeat or collide with an earlier attempt.
- Close a failed prepared run with `tools/guardian_tuning_evidence_builder.py abort`. The command writes immutable failure evidence, preserves the no-repeat digest, clears the crashed contract, and requires a materially changed review before retry.
- Artifact byte/sample/queue bounds, content-addressed filenames, execution provenance after `prepared_at_utc`, and fresh-read evidence-chain validation are mandatory. Tuning evidence never grants trade permission; it can only tighten the named technical floor, while the verifier, risk engine, and gateway still decide each order.
