## 2. Operator Model

The trader is a single role with one playbook (`docs/SKILL_trader.md`). The scheduled task picks which model executes a given cycle; the playbook, the gateways, and the broker account are identical regardless of which model runs. There is no "Codex trader" vs "Claude trader" — there is **the trader**, and a scheduler that dispatches it.

Rules:

- **Exactly one trader scheduled task enabled at a time.** Two tasks against the same OANDA account = duplicate orders, double-counted exposure, contradictory protection actions.
- **DecaBot is separate from the main trader scheduler.** DecaBot lives at `/Users/tossaki/App/DecaBot`, uses its own OANDA account and launchd agents, and is operated from this repo only through the operator bridge `scripts/qr-decabot.sh` plus the weekend switcher's launchd pause/restore of DecaBot labels. It must not be counted as a second QuantRabbit scheduler against the main account, and QuantRabbit live code must not import or route DecaBot order writes.
- **Switch by enabling / disabling scheduled tasks.** No code, prompt, or playbook changes are required to swap the executing model.
- **Weekend scheduler guard.** Codex automation `qr-weekend-market-off` pauses related QuantRabbit schedulers every Saturday 06:00 JST; `qr-weekend-market-on` restores them every Monday 07:00 JST. The switcher snapshots current task state before pausing and restores only that snapshot, so a disabled Claude trader is not enabled accidentally and multiple trader schedulers are never restored. By default it manages weekday QuantRabbit Codex automations discovered under `~/.codex/automations` (`qr-*`, excluding `qr-weekend-*`) plus weekday Claude scheduled tasks whose project is `QuantRabbit` or `QuantRabbit-live`; explicit `QR_WEEKEND_CODEX_TASKS` / `QR_WEEKEND_CLAUDE_TASKS` env lists override discovery. `qr-self-improvement-watch` is pause-managed but never restore-enabled: Monday restore must force it to `PAUSED` even if an older Saturday snapshot recorded `ACTIVE`, so the repair agent cannot turn itself back on automatically. It also snapshots and stops DecaBot launchd labels `com.decabot.ai`, `com.decabot.monitor`, and `com.decabot.review`, then restores only labels that were loaded in the snapshot; `com.decabot.live` remains unmanaged unless the operator explicitly requests it. While `~/.codex/quant_rabbit_weekend_task_state.json` has `mode=paused`, `sync-live-runtime.sh` may accept `qr-trader` as `PAUSED`; without that guard state, a paused trader automation remains a sync blocker.
- **Every cycle reads the same `docs/SKILL_trader.md`.** Changes there affect every subsequent cycle whichever model runs.
- **Runtime/development worktrees are separated.** Active trader scheduled tasks must use `/Users/tossaki/App/QuantRabbit-live`, a clean committed worktree dedicated to broker reads, receipts, and gateway execution. Development and strategy improvement may happen in `/Users/tossaki/App/QuantRabbit` or another worktree, but a dirty development tree must never be the live trader `cwd`.
- **Runtime health checks are local-live first.** Current production inspection uses `/Users/tossaki/App/QuantRabbit-live` artifacts: `data/broker_snapshot.json`, `logs/trader_journal.jsonl`, `docs/*_report.md`, and live/runtime git HEAD. Do not introduce external-host or process-manager checks unless the operator explicitly requests that environment.
- **Main is the stable integration branch.** Development commits are promoted by `scripts/sync-live-runtime.sh`: source branch → `main` (fast-forward only) → `codex/live-trader-runtime` (fast-forward only). Do not commit directly in `/Users/tossaki/App/QuantRabbit-live`; that branch mirrors `main` for runtime and creates no live-side commits. The live wrapper runs `scripts/sync-live-runtime.sh --live-only --skip-tests` before each gateway cycle so a promoted `main` cannot be forgotten and runtime `docs/*_report.md` drift is cleared.
- **Promotion is automated and guarded.** The development post-commit hook installed by `scripts/install-live-runtime-hooks.sh` runs the sync script after each commit. Promotion is blocked by source/config/data/decision dirt, non-fast-forward history, failing tests, missing live worktree, or a scheduled trader automation that does not point at `/Users/tossaki/App/QuantRabbit-live`. Report-only `docs/*_report.md` drift may be cleared automatically in the live worktree because the next trader cycle overwrites latest reports.
- **Improvement work must finish in live.** Any agent that implements, repairs, tunes, or documents behavior that affects the trader runtime must not stop at the development worktree. Before ending the session, the agent must commit the logical change, let the guarded promotion path run, and verify that `/Users/tossaki/App/QuantRabbit-live` is at the promoted commit. If promotion is blocked, the agent must report the exact blocker and leave a committed source branch ready to promote; uncommitted improvements are not considered done. Only an explicit operator instruction to hold back a change may skip live promotion.
- **Precheck before report writes.** Concurrency and clean-source checks must run before any command that writes tracked reports under `docs/*_report.md`. Dirty source/config/data/decision files block the cycle. Report-only tracked diffs under `docs/*_report.md` from the immediately previous trader cycle are runtime drift and do not block a scheduled run; the next cycle may overwrite latest reports. A stopped precheck must not create new diffs; otherwise the next scheduled cycle will self-block before it reaches the market read. A recent decision receipt is not a stop condition by itself: the router must distinguish unconsumed receipts from receipts already rejected, already handed to a gateway, or made stale by a newer broker snapshot / repriced intent packet.
- **Handoff discipline.** When the active scheduled task changes, the next cycle must (1) refresh `broker-snapshot`, (2) read the most recent `data/codex_trader_decision_response.json` (the filename is kept for compatibility, not as an attribution), (3) inherit any open trader-owned positions and pending orders, (4) not cancel another cycle's pending orders without an explicit reason recorded in the next decision receipt. When a scheduled trader has already generated `data/broker_snapshot.json` and `data/order_intents.json` before writing the decision receipt, the gateway cycle must use `autotrade-cycle --reuse-market-artifacts` so GPT verification sees the same evidence packet the decision cited; `LiveOrderGateway` still fetches fresh broker truth immediately before staging or sending.
- **Broker truth reconciliation must continue the cycle.** If refreshed broker truth says old local state, journal text, pending-order claims, or a prior receipt are stale, broker truth wins and the trader must re-route, rewrite the decision when needed, and continue through the gateway path. Absence of a locally remembered pending order in the refreshed OANDA snapshot is not a no-trade gate. Only an active live lock, another enabled trader scheduler, dirty source/config/data/decision files, missing required credentials, or actual current OANDA broker-truth exposure/risk gates may stop the cycle before decision work.
- **Position guardian yields to the full trader cycle.** The launchd `position-guardian` exists to catch short-lived profit-protection and TP-progress exits between full trader cycles. It must use the shared live lock and skip with a successful no-op when a full trader/runtime cycle already owns the lock, so high-frequency profit capture cannot starve entry refresh, broker reconciliation, or decision verification. The full live wrapper may wait briefly for an in-flight guardian to finish; a newly fired guardian must not wait behind or compete with the full trader cycle. Before loading or reloading the launchd agent, `scripts/install-position-guardian.sh --check` must pass against `/Users/tossaki/App/QuantRabbit-live` so activation cannot start from an unsynced runtime, non-report dirty worktree, missing guardian script, or invalid OANDA env file.
- **Position guardian active-state gate for fresh entries (2026-06-22).** `scripts/run-autotrade-live.sh` and live CLI defaults set `QR_REQUIRE_POSITION_GUARDIAN_ACTIVE=1`; the wrapper reads guardian health with `scripts/install-position-guardian.sh --require-loaded` before a live `--send` cycle, while direct CLI sends fall back to `LiveOrderGateway`'s read-only launchd/plist/heartbeat check when `QR_POSITION_GUARDIAN_ACTIVE` was not exported by the wrapper. `--require-loaded` and the direct fallback must prove both the launchd label and a fresh heartbeat from `data/position_guardian_execution.json` or `data/position_guardian.json` (default freshness window `QR_POSITION_GUARDIAN_INTERVAL * 4`, overrideable by `QR_POSITION_GUARDIAN_HEARTBEAT_MAX_AGE_SECONDS`); a loaded but stale/failing guardian is still inactive for fresh-entry purposes. The wrapper must not stop the full cycle when the guardian is missing or stale, because existing-position TP/protection work still has to run; instead it exports `QR_POSITION_GUARDIAN_ACTIVE=0` and `LiveOrderGateway` blocks only fresh entry sends with `POSITION_GUARDIAN_INACTIVE_FOR_SEND`. This is the executable repair for the USD_JPY 472792 miss where TP-progress profit became a stop-loss while the fast guardian was not loaded. `self-improvement-audit` must also surface inactive required guardian state as `POSITION_GUARDIAN_INACTIVE_FOR_PROFIT_CAPTURE` so the next route treats the defect as "why plus P/L was not banked", not as an entry-resend/cancel issue. Escape condition: the operator may explicitly set `QR_REQUIRE_POSITION_GUARDIAN_ACTIVE=0`, which is an auditable override, not the default; `QR_POSITION_GUARDIAN_REQUIRE_HEARTBEAT=0` is also an auditable operator override for the heartbeat portion only and must not be the live default.
- **Trader support bot operator panel (2026-06-22).** `python3 -m quant_rabbit.cli trader-support-bot` is a read-only support command that writes `data/trader_support_bot.json` and `docs/trader_support_bot_report.md`. It must not send orders, close positions, cancel entries, or load launchd agents. Its job is to show the trader whether guardian heartbeat/launchd proof is green, whether TP-progress profit capture was missed, whether self-improvement/profitability P0s are blocking turnover, and which repair-frontier lanes would still have non-guardian blockers after support services recover. Any action that changes live services, especially loading/reloading `position-guardian`, remains an explicit operator action after `scripts/install-position-guardian.sh --check` passes.
- **Specialist support bots (2026-06-22; repair orchestrator added 2026-06-23).** The trader may add as many specialist support bots as the current failure mode requires, but support bots are read-only by default and must publish their input artifacts, output JSON/report paths, status, blockers, operator actions, and `live_side_effects=[]`. A support bot may diagnose, rank, or propose code/ops work; it must not mutate live services, send/cancel orders, or close/modify positions unless the relevant gateway, explicit live flags, tests, commit, and guarded runtime sync already authorize that separate execution path. `profit-capture-bot` is the first specialist: `python3 -m quant_rabbit.cli profit-capture-bot` writes `data/profit_capture_bot.json` and `docs/profit_capture_bot_report.md`, recalculates the current TP-progress profit-capture gates for each open trader-owned position, and explains whether it is `BANKABLE_NOW`, watch-only, or blocked by missing quote/chart/ATR/TP inputs. `trader-support-bot` reads this specialist output so the cycle digest shows both historical TP-capture misses and the current open-position capture trigger. `trader-repair-orchestrator` reads `trader-support-bot.repair_requests` and writes a Codex-executable repair queue plus a top-level `codex_work_order`: Codex may read artifacts, edit code/tests/runtime contract docs, run tests, commit, and sync live; order sends, order cancels, position closes, and launchd load/reload remain explicit-approval or existing-gateway-only actions. If a partially refreshed or older support artifact is `SUPPORT_BLOCKED` but has no top-level `repair_requests`, the orchestrator must rebuild the queue from embedded support evidence such as `profitability_acceptance.repair_plan`, guardian state, and repair-frontier blockers instead of reporting `NO_REPAIR_REQUESTS`. The orchestrator is read-only, grants no live permission, and QuantRabbit code still must not call model APIs.
- **Repair queue dependency ranking (2026-06-23).** `trader-repair-orchestrator` must rank repair work by live profit dependency, not alphabetic P0 order or loose keyword ties: direct TP-progress profit-capture repair comes before close-gate evidence persistence, both come before month-scale residual entry-quality scaling, and evidence-window-only historical findings such as already-persisted BLOCK close evidence are `WAITING_FOR_LIVE_EVIDENCE_WINDOW` rather than `READY_FOR_CODEX_IMPLEMENTATION`. A trader's specific request may select a lower dependency repair only when it has a stronger textual match; tied matches fall back to dependency rank.
- **Repair queue dependency waits are not code work (2026-06-23; live-evidence wait clarified 2026-06-24).** If `REPAIR_TP_PROGRESS_PROFIT_CAPTURE_REPLAY` is backed by the current production-gate replay contract but the evidence also says `guardian_profit_capture_inactive=true` / `TP_PROGRESS_REPAIR_REPLAY_NOT_DEPLOYED`, the orchestrator must classify that request as `WAITING_FOR_OPERATOR_APPROVAL` through `RESTORE_POSITION_GUARDIAN_AFTER_PREFLIGHT` instead of selecting it as Codex code implementation work. Codex may then work the next actionable code/evidence repair, such as month-scale residual entry-quality or forecast-frontier blockers, while launchd load/reload and the later live evidence window remain explicit-approval or live-runtime proof tasks.
- **Residual entry-quality loop break (2026-06-23).** If `REPAIR_MONTH_SCALE_RESIDUAL_ENTRY_QUALITY` is raised while current `order_intents` already show matching month-scale residual blocker codes (`MONTH_SCALE_ENTRY_QUALITY_RESIDUAL_BLOCKED` / `MONTH_SCALE_RESIDUAL_LOSS_REPAIR_BLOCKED`), the support/orchestrator loop must classify it as a replay/evidence-window wait rather than another Codex implementation task. The next executable step is the documented `execution-timing-audit --lookback-hours 744 --post-close-hours 6` plus `profitability-acceptance` refresh, or the next unblocked frontier repair; re-implementing the same residual block is a loop bug.

Correct guardrails that block low-payoff, sell-the-low, or buy-the-high shapes are not Codex code repair work unless a failing regression proves a false positive and the original invalid chase remains blocked; otherwise they are `FRONTIER_PROTECTIVE_GUARDRAIL_ACTIVE` live-evidence waits, not gate-loosening tasks. `MARGIN_TOO_THIN_FOR_MIN_LOT` is also not gate-loosening work: the 1000u production floor remains a broker/spread safety invariant, so repair-frontier rows blocked only by current free-margin/risk-budget capacity must become `FRONTIER_MARGIN_CAPACITY_WAIT` with a clearance condition to refresh broker truth after margin frees or approved exposure changes. Do not lower `MIN_PRODUCTION_LOT_UNITS` or emit sub-1000u live receipts to make a repair lane executable.
- **Directional inversion counterfactual loop break (2026-06-24).** A losing broker-truth position whose gross opposite-side P/L would clear the daily 5 % floor is a P0 repair signal, not live inversion permission. `trader-support-bot` must mark that repair as `WAITING_FOR_DIRECTIONAL_INVERSION_REPLAY_EVIDENCE` until repeated spread-included inversion evidence exists, such as a matching `qualified_inversion_selectors` row. If the current pair-filtered bid/ask replay instead has no contrarian support and a pair-level negative rule, the same gross counterfactual is `CONTRARIAN_REPLAY_REJECTED` and must not keep re-entering the Codex repair queue; the loop must move to the next actionable evidence/repair item. Unit tests must not implicitly read repo/live replay artifacts when they use a temporary output path.

In automation, "GPT" means whichever model is currently dispatched by the scheduler. **Do not call any API-key model path from QuantRabbit code** (`QR_OPENAI_API_KEY`, `OPENAI_API_KEY`). Live discretion lives in the scheduled-task model, not in a library call.

## 3. Prime Directive

- Build from broker truth outward.
- Enforce broker gateway and risk contracts in code before prompt or strategy-prose optimization.
- Express discretionary autonomy as executable receipts, not vague market prose.
- Default to dry-run / read-only behavior unless live gates are explicitly satisfied.
- **Adapt to live market conditions, not to fixed numbers.** Sizing, stop distance, target distance, spread tolerance, exposure budget, and pacing must all be derived from current ATR, regime, spread, equity, daily progress, and liquidity — not from JPY / pip / multiplier literals baked into code or prompts.

## 3.1. Critical Thinking / Anti-Absolutism

Every QuantRabbit guard, profile status, prompt rule, score penalty, audit blocker, and safety wrapper is a falsifiable hypothesis, not absolute truth. A rule created to repair one failure can become the next profit blocker. Agents must therefore audit self-created mechanisms with the same skepticism applied to market forecasts.

- Treat every blocker as guilty until it proves current value: when profitability is poor or opportunity coverage is thin, rank `live_blocker_codes`, cancel regrets, missed MFE, stale artifacts, and exit-reason drag by actual opportunity cost before adding another rule.
- A guard must publish its escape condition in executable form. If prose says "require a new vehicle or market-structure proof" but code never recognizes that proof, the guard is a bug.
- Prefer downgrade paths over permanent bans: use `WARN`, smaller size, non-market triggers, shorter TTL, attached TP, or explicit evidence collection when broker truth, current forecast, rail geometry, and gateway risk checks still support a bounded attempt.
- Negative history is advisory unless it is current, well-sampled, and still matches the present vehicle. Do not let stale or over-segmented strategy memory block a fresh market-proof `LIMIT` / `STOP-ENTRY` that has current forecast support, correct structure, bounded risk, and attached TP.
- Before preserving a blocker, state the strongest counterargument against it. If the counterargument is supported by live ledger, execution-timing audit, or capture economics, change the blocker or make it evidence-conditional.
- New guards need two tests: one proving the original failure remains blocked, and one proving the guard does not block the valid current shape it claims to allow.

## 3.5. No Thoughtless Hardcodes or Fallbacks

This rule applies to every file the trader reads or writes — `src/quant_rabbit/**`, `tools/**`, CLI flags, scheduled-task SKILLs, prompts, and config.

- **Hardcoded JPY / pip / multiplier values are forbidden** unless they are documented, named (`InstrumentSpec.normal_spread_pips`, `RiskPolicy.daily_risk_pct`, etc.), and clearly marked as a *test default* or a *broker-spec constant* — not as a production risk decision.
- **Silent fallbacks to a literal are forbidden.** If a required input is missing, raise (or emit a `*_MISSING` issue with severity `BLOCK`) so the operator sees it. Falling through to `RiskPolicy().max_loss_jpy` or `8 pips` or `1.5 R` quietly is the failure mode this contract was written to prevent.
- **Risk caps must be equity-derived, and per-trade ≠ per-day.** The daily-target ledger (`start_balance × daily_risk_pct`) is the **whole-day** budget. The **per-trade** cap is `daily_risk_budget_jpy / target_trades_per_day` — this is what flows into `intent.metadata['max_loss_jpy']`. The split is mandatory: a single losing trade must not be able to exhaust the day's budget, because the campaign needs many attempts to secure the 5% floor and, on extension days, reach the 10% target. `target_trades_per_day` must come from `ai_test_bot_backtest.firepower.required_trades_per_day_at_observed_expectancy` when available; `RiskPolicy.target_trades_per_day = 10` is only the no-evidence fallback. Override via `--target-trades-per-day` on `daily-target-state` for an explicit operator pace. Never derive either cap from a JPY literal stored in code or in a stale state file.
- **Geometry must be ATR-derived.** SL distance and TP distance must scale with current ATR (per pair, per timeframe, per regime). `spread × N` is a *floor*, not the geometry. If pair-charts data is unavailable, the cycle waits — it does not invent numbers.
- **Reward / risk targets must be regime-derived.** A single fixed `target_reward_risk` for all conditions is the same anti-pattern as a fixed loss cap. Trend regimes deserve wider targets; range regimes deserve faster rotation.
- **Spread tolerance must be liquidity-derived.** A static `2.5 ×` multiplier across all sessions ignores Tokyo open vs London close. Tighten in liquid hours, loosen in thin hours, and BLOCK above session-aware caps.
- **No "default-to-yesterday" stale persistence.** State files (e.g. `data/daily_target_state.json`) must invalidate on a new campaign day or when the snapshot post-dates the file by more than one cycle. Reading a stale value and pretending it is current is a silent fallback in disguise.
- **Every numeric constant in production code requires a docstring or comment** stating: (a) what market reality it represents, (b) why it is constant rather than market-derived, and (c) what should replace it if it ever needs to be changed. If you cannot write that comment, the value should not be a constant.
- **When in doubt, fail loud.** A `BLOCK` issue with a clear message is always preferable to a guess. The operator can act on a blocker; the operator cannot recover from a silent miscalculation.
- **Minimum production lot floor (`MIN_PRODUCTION_LOT_UNITS = 1000`, 2026-05-12).** When the margin headroom shrinks so the loss-budget or NAV-pct sizing path resolves to fewer than 1000 units, the trader must NOT emit a fillable receipt. Sub-1000-unit FX orders cannot capture a pip target larger than the round-trip OANDA spread, so the trade is structurally guaranteed-loss before market direction even matters. The floor is enforced at two layers: (i) `intent_generator._risk_budgeted_units` returns `0` for sub-floor results, and the build-intent caller emits a `MARGIN_TOO_THIN_FOR_MIN_LOT` BLOCK so the intent becomes DRY_RUN_BLOCKED; (ii) `RiskEngine.validate` independently flags `0 < |units| < MIN_PRODUCTION_LOT_UNITS` with `MIN_LOT_VIOLATION` BLOCK, providing defense-in-depth for manual `stage-live-order`, replayed legacy receipts, and other paths that bypass intent_generator. The 1000-unit floor matches the existing `_risk_budgeted_units` rounding granularity and the OANDA retail FX micro-lot trade size, so it is a broker-policy constant, not a tunable market-derived threshold. Test fixtures that exercise legacy sub-1000-unit edge cases must opt out via `QR_ALLOW_TEST_MICRO_LOT=1`; production code must never set this env. The 2026-05-12T07:46 UTC sequence (470901 EUR/USD SHORT 201u + 470904 AUD/JPY SHORT 322u + 470907 GBP/USD SHORT 2u, total -4 JPY but spread cost guaranteed loss) is the regression this floor prevents.
- **Same-pair hedge margin uses longest-leg increment (broker truth, 2026-05-19; clarified 2026-05-26).** On OANDA v20 hedging accounts, same-instrument opposite exposure is margined by the larger side, not by gross long + short notional. QuantRabbit must therefore estimate broker margin increment from the account-wide same-pair long/short legs, including operator-managed manual/tagless trades, because that is what OANDA uses for `marginUsed`: a 1k EUR/USD SHORT against an account with 22k EUR/USD LONG and 8.4k SHORT has `estimated_margin_jpy=0` until the short side exceeds the 22k long leg. This broker-truth margin offset is separate from strategy hedge authorization. Explicit `metadata['position_intent']='HEDGE'` sizing and `HEDGE_UNITS_EXCEED_OPPOSITE_EXPOSURE` still use trader-owned opposite exposure only, so manual/tagless exposure cannot silently authorize an oversized bot hedge. Non-hedge pyramids may still have zero incremental broker margin when they add to the account's smaller side, but they must pass normal strategy, portfolio, spread, quote-freshness, and open-position gates. `RiskEngine`, `intent_generator._risk_budgeted_units`, and `LiveOrderGateway` basket checks must all use the same broker incremental-margin value so a margin-neutral same-pair add does not fall to `MARGIN_TOO_THIN_FOR_MIN_LOT` or `MARGIN_UTILIZATION_CAP_EXCEEDED`.
- **Hedge sizing follows the uncovered margin-free opposite leg.** In SL-free runtime, an explicit same-pair hedge targets the current margin-free opposite leg before NAV% sizing, so a 22k LONG can generate a 22k SHORT hedge instead of a 7k NAV-sized partial. If 5k SHORT already exists against that 22k LONG, only 17k remains hedgeable; if 22k SHORT already exists, the generator suppresses the recovery-hedge label and `RiskEngine` blocks any hand-written HEDGE with `HEDGE_REFERENCE_ALREADY_COVERED` or `HEDGE_UNITS_EXCEED_OPPOSITE_EXPOSURE`.
- **Trapped-position recovery hedge is not a fresh chase.** When the same-pair opposite leg is underwater, `intent_generator` marks the hedge with `metadata['hedge_recovery']=true`. That hedge is allowed to monetize the move against the trapped exposure, but only when timing is not chasing the already extended side: non-market pullback/retest/triggered recovery context can downgrade 24h exhaustion-chase and opposing reversal-pattern evidence to WARN, while same-side exhausted `MARKET` recovery remains BLOCK. A recovery hedge must wait for a pullback, retest, or non-market trigger instead of buying/selling the already extended price. `RiskEngine` uses the range/recovery reward floor (`range_min_reward_risk`) rather than the fresh trend-entry floor. This does not waive spread economics or bad geometry: `TARGET_TOO_THIN_FOR_SPREAD`, stale quote, invalid entry side, non-hedge adds, and missing broker hedging proof still block.
- **Hedge timing classes and unwind discipline (2026-05-20).** A same-pair hedge intent must publish `metadata['hedge_timing_class']`: `LOCK_GAIN` for profitable existing exposure, `REVERSAL` for underwater exposure with machine-readable reversal confirmation, or `CONTINUATION` for underwater exposure still moving against the original side without reversal proof. Every hedge must also publish `metadata['hedge_unwind_plan_required']=true` and a `hedge_review_trigger` so the trader treats it as a time-boxed revenue/risk leg, not passive loss-freezing. `RiskEngine.validate` blocks HEDGE intents missing these fields so manual `stage-live-order`, replayed receipts, and gateway paths cannot bypass the discipline. Recovery continuation hedges are capped below the neutral recovery tranche (`RECOVERY_HEDGE_CONTINUATION_MAX_SCALE = 0.35`, mirrored as `risk.HEDGE_CONTINUATION_MAX_SCALE`) because they are low-quality momentum chases; they may monetize urgent adverse movement, but must not freeze the full trapped leg at a likely exhaustion point. Reversal and lock-gain hedges can use larger market-derived tranches when structure, TF agreement, forecast direction, spread economics, and margin rules pass.
- **Broker-side SL rollback + precision filters (A/B/C/D/E, 2026-05-13).** The 2026-05-12T15:33 UTC mass-close incident (-22,980 JPY confirmed loss on 5 protected positions) showed that an honor-system Gate B (`operator_close_authorized` JSON field) is not a structural protection: a panic-close trader can self-set the field and the verifier passes. The first repair attempted broker-side SL on every NEW entry, but live feedback (`feedback_broker_sl_noise_hunt.md`, `feedback_no_tight_sl_thin_market.md`) showed that even widened broker SLs were still harvested by thin-session noise. (Amended 2026-06-11 by K below: a non-trailing catastrophe-distance disaster stop is attached to new entries by default — it bounds dislocation tails, not trade noise.)

  - **B — price percentile context.** `chart_reader._build_extended_confluence` publishes `price_percentile_24h` / `price_percentile_7d` / `atr_percentile_24h` (0.0-1.0, market-derived from H1 / H4 close distributions). `trader_brain._apply_directional_gating` penalises same-side entry at the 95th/5th percentile by `PRICE_PERCENTILE_EXTREME_PENALTY (= 25.0)` and rewards mean-reversion entries at the same extreme by `PRICE_PERCENTILE_MEAN_REV_BONUS (= 15.0)`. The 0.95/0.05 boundaries are the standard 5%/95% distribution extremes used throughout statistics, not tuned market literals.

A range/failure receipt waiting on the M5/M15 structural retest side is not a chase: SHORT upper-half retest/rejection and LONG lower-half retest/rejection use the same box midpoint geometry as breakout-failure timing, so they remain eligible while sell-the-low / buy-the-high MARKET or STOP entries are blocked.

This is NOT the noise-band trading stop the 「SLいらない」 directive banned: the 2026-05-13 hunts were 4-42 pip stops; this bound is 60-120+ pips on majors and only fires on true dislocations (flash moves, interventions, weekend gaps) that strike inside the trader's 20-minute blind window — the class of event that produced the 2026-05-14 `MARKET_ORDER_MARGIN_CLOSEOUT` forced liquidations (-5,641 JPY) and the unbounded give-up-close tail (avg -1,437 JPY vs avg win +376). Three invariants keep it from recreating past failures: (i) it is computed separately from `intent.sl`, so sizing, reward/risk, and risk validation are unchanged — sizing against the disaster distance would recreate the micro-lot spiral; `LiveOrderGateway` must not shrink individual order units merely because the catastrophe stop is farther than the per-trade expected-invalidation cap; (ii) it never trails (`QR_DISABLE_TRAILING_SL=1` stays on; the 2026-05-13 incident was trailing tightening into noise, not the initial placement); (iii) existing positions are never retro-fitted (existing-position invariant). When H4 ATR evidence is missing the entry proceeds exactly as the pre-K SL-free runtime did and `metadata['disaster_sl_missing']` surfaces the gap — no silent substitute distance (§3.5). One intended second-order effect: pending-order basket risk is read from the broker order's real `stopLossOnFill`, so the portfolio's SIMULTANEOUS catastrophe exposure is now naturally bounded by the daily risk budget (≈2-3 concurrent NAV%-sized positions); if the full catastrophe-stop risk cannot fit the remaining basket/portfolio capacity, the lane is blocked instead of micro-lotted. That bound is the §5 risk model working, not a regression; widen it only by intentionally raising `daily_risk_pct`. Knobs: `QR_DISASTER_SL` (default 1 in the SL-free bootstrap), `QR_DISASTER_SL_H4_ATR_MULT`.

This rule is enforceable: any reviewer (Codex or Claude) seeing a JPY literal, a `pip` constant, a fixed multiplier, or a silent `or default_value` in trader code MUST flag it before merging.

## 6. Autonomous Pro Trader Requirements

- The operator is the discretionary decision layer, but every decision must be grounded in structured broker state, mined history, current market context, and risk geometry.
- The trader must decide whether to TRADE, WAIT, CANCEL, PROTECT, TIGHTEN, CLOSE, or REQUEST_EVIDENCE.
- **No-trade is a valid professional decision** when edge, spread, timing, liquidity, narrative, or risk / reward is inadequate — but it must be defensible against the campaign, not against an invented threshold.
- **WAIT discipline.** A WAIT decision is only valid when it cites a specific gate that fired, by name from this contract or from `data/`. Generic prose ("Golden Week thin liquidity", "EVENT_RISK", "UNCLEAR regime") is not sufficient. When `daily_target_state.json` shows `progress_pct < 50` AND `data/order_intents.json` lists ≥ 3 `LIVE_READY` lanes, WAIT additionally requires (a) one chart-story sentence per LIVE_READY lane stating why **that lane's specific invalidation** is hit right now, citing M5 numbers from `pair_charts.json`, and (b) explicit citation of the contract gate (e.g. §9 spread cap, §11 strategy block) or the `*_BLOCK` issue from `risk-dry-run`.
- **No invented thresholds.** The trader must not stack additional risk gates beyond those listed in §3.5, §8, §9, §10, §11. Examples of forbidden invented gates: "ATR×2 safety floor for thin markets", "need 2× normal spread before entry", "skip all trades during a holiday week", "regime must be CLEAR before entry". When a condition feels risky, **size it down** — `per_trade_risk_budget_jpy` already shrinks the per-shot exposure — do not block the lane in prose.
- **Precedent must be rescaled.** Citing a past loss (e.g. "Apr 3 -984 JPY") to justify a current decision requires rescaling the precedent to **today's** per-trade cap: `precedent_loss × (current_per_trade_cap / cap_at_time_of_precedent)`. A precedent recorded at per-trade=4000 has 1/10 the weight under per-trade=400. Cite the rescaled figure or do not cite the precedent.
- Historical worst loss is a repair warning before current risk geometry is proven, not a second hard veto. Once the current `LIVE_READY` receipt fits `per_trade_risk_budget_jpy`, old loss memory becomes audit context only.
- Past negative evidence must not harm a current `LIVE_READY` lane. Negative `live_net`, missing positive mined evidence, low capture rate, old `WAIT` language, or stale story rejection markers may be printed as advisory context, but they must not create blockers, reduce size, or reduce rank after current broker-truth risk geometry and strategy profile validation have already produced `LIVE_READY`.
- Every trade decision compares historical analogs, current regime, market story, chart structure, campaign role, open exposure, pending orders, invalidation distance, and the short / medium / long horizon opportunity map (M1/M5 execution, M15/M30/H1 operating swing, H4/D anchor). The trader must not collapse the market into one timeframe: a protected runner can keep its TP while another pair or another horizon presents a separate risk-bounded entry.
- **20-minute decision depth (2026-06-05).** The live trader cadence is about one discretionary decision every 20 minutes, so `TRADE`, `WAIT`, and `REQUEST_EVIDENCE` receipts must include `twenty_minute_plan`: `horizon_minutes=20`, the primary path expected before the next cycle, the failure path, the entry/hold trigger, the invalidation/cancel trigger, the strongest counterargument, the next-cycle check, and packet evidence refs. This is a receipt-depth requirement, not a new market-risk gate: it must force deeper reasoning from the existing broker/market packet without inventing extra thresholds.
- Every decision receipt states: selected lane, rejected alternatives, thesis, method, narrative, chart story, invalidation, TP, SL, units, expected reward, worst-case loss, reason for action, and `operator`.
- The system separates observation, reasoning, risk validation, order staging, order sending, position protection, and post-trade learning.
- The trader learns from wins, losses, missed edges, bad entries, premature exits, and protection repairs **without allowing memory to override live risk checks**.
- Maintain a professional audit trail: broker snapshots, intents, dry-run receipts, live-stage receipts, execution reports, protection actions, post-trade reviews, and the append-only execution ledger.
- `execution-ledger-sync` records OANDA transaction truth into `data/execution_ledger.db`. `stage-live-order` and `autotrade-cycle` sync it before and after gateway work and record gateway receipt files, so entries, exits, protections, cancels, and broker-side transaction ids have one durable DB trail.
- Continuously root-cause bugs in strategy logic, risk math, stale broker state, duplicated exposure, missing protection, and legacy assumptions.

- **Directional gating + margin-aware basket (C-1 / C-2 / C-4, 2026-05-12).** The trader_brain `_apply_directional_gating` pass, run between `_score_lane` and basket construction, reshapes the NEW-entry lane set when bias and attack_advice agree on direction. It never touches `PositionManager` / `PositionProtectionGateway` / existing-position state — only the LaneScore tuple feeding `_basket_lane_plan`.
  - **C-1 directional gating.** For every pair appearing in scores, if `pair_charts[pair].confluence.score_balance` is `LONG_LEAN` or `SHORT_LEAN` AND `|score_gap| >= DIRECTIONAL_GATING_STRONG_GAP` (= `2 ×` the documented `chart_reader` TIED noise floor `0.05`, encoded as `DIRECTIONAL_GATING_STRONG_GAP_MULTIPLIER`) AND the `ai_attack_advice` top-K majority direction for the same pair agrees with the bias, every `LIVE_READY` lane in the OPPOSITE direction is demoted from `SEND_ENTRY` to `NO_TRADE` with a `directional_gating_demoted` blocker.
  - **C-2 attack_advice directional veto.** Independently of pair_charts bias, when the top-K advice for a pair has a strict majority in one direction, lanes in the opposite direction lose `ATTACK_ADVICE_VETO_PENALTY = 25.0` score points and gain an `attack_advice_veto` rationale entry. The magnitude matches the existing `-25` penalty grid for direction-conflict downgrades, so the veto remains internally consistent.
  - **C-4 margin-aware basket truncation.** `_basket_lane_plan` and `_expanded_gpt_basket_plan` track cumulative `LaneScore.estimated_margin_jpy` (sourced from `intent_generator.risk_metrics.estimated_margin_jpy`) and stop adding lanes once cumulative margin would exceed the gateway-equivalent effective room `min(snapshot.account.margin_available_jpy, snapshot.account.nav_jpy × RiskPolicy.max_margin_utilization_pct - snapshot.account.margin_used_jpy) × MARGIN_AWARE_BASKET_BUFFER` (= 0.9, documented engineering buffer for intra-cycle drift). This prevents the basket from over-claiming margin and letting the gateway reject every candidate at staging with `BASKET_MARGIN_UTILIZATION_CAP_EXCEEDED`. The buffer is engineering tolerance, not a market-derived threshold — comment in code states the rationale.
  - **Existing-position invariant.** None of C-1 / C-2 / C-4 reads `snapshot.positions`, `snapshot.orders`, or any TP/SL field. Position management remains the exclusive domain of `PositionManager` + `PositionProtectionGateway`, which run against a separate `managed_snapshot` and a separate `position_decision` flow. The directional gate cannot — by construction — alter SL/TP/CLOSE behavior on open trades.
- Strategy-review identity is `lane_id` plus `method`, not a loose desk alias. A review for `TREND_CONTINUATION` cannot authorize a `RANGE_ROTATION` or `BREAKOUT_FAILURE` lane.

- Real entry sends require **all** of:
  - `QR_LIVE_ENABLED=1`
  - `--send`
  - Fresh broker truth (recent `data/broker_snapshot.json`)
  - Explicit lane id
  - `RiskEngine.validate(..., for_live_send=True)` passes
  - Strategy-profile validation passes

Existing-position TP/close/protection routing has higher priority and must not be delayed by stale entry feedback.
- While the daily target is still open, an empty, malformed, missing, or history-DB-stale `data/strategy_profile.json` is also a refresh blocker before target-open entry / TRADE-verify routing. The refresh branch must rebuild it with `import-legacy` + `mine-strategy`; otherwise every lane can look like `STRATEGY_PROFILE_MISSING` even when the archive DB contains usable evidence. This is evidence repair, not permission to bypass forecast, risk, spread, or strategy validation.
- While the daily target is still open, `data/memory_health.json` must also be current and non-blocked before target-open entry / TRADE-verify routing. `memory-health` aggregates short-term broker/order/capture-economics/forecast memory, medium-term projection/execution/learning memory, long-term strategy_profile memory, and position entry_thesis coverage. It must record the `data/capture_economics.json` timestamp it audited and block when `capture-economics` was refreshed after `order_intents`, or when intent metadata embeds a stale capture-economics trade count. A blocked audit routes back to refresh; it does not grant permission to bypass RiskEngine, IntentGenerator telemetry validation, or LiveOrderGateway.
- While the daily target is still open, `data/profitability_acceptance.json` must be current before target-open entry / TRADE-verify routing. Missing, unreadable, or stale acceptance routes back to `refresh_market_context` after `self-improvement-audit`; P0 acceptance findings route to learning/repair and block high-turn entry scaling. The only entry exception is a current `LIVE_READY` lane that explicitly carries the attached-TP HARVEST repair metadata (`self_improvement_p0_repair_live_ready=true`, `self_improvement_p0_repair_mode=TP_HARVEST_REPAIR`), and even then the receipt may trade only that repair basket. Acceptance is the red/green profit invariant gate; it must not remain a report that the router ignores.

- Use explicit env vars: `QR_OANDA_TOKEN`, `QR_OANDA_ACCOUNT_ID`, `QR_OANDA_BASE_URL`.

- `OandaReadOnlyClient` loads `.env.local` automatically when process env vars are absent. `QR_OANDA_ENV_FILE=/path/to/file` overrides the lookup for tests or alternate environments.
- Do **not** read legacy `config/env.toml` into vNext.
- Manual/tagless operator exposure does **not** block new trader entries. TP-only profit management may modify manual/tagless take-profit orders or close an already-profitable partial, but the trader must not add/replace SL, trigger loss-side partial closes, or market-close manual/tagless positions. External/broker-synced exposure that is not explicitly operator-managed still blocks new entries until adopted or closed.

## 10. Position Protection

- Missing SL repair remains a **repair requirement** when SL mode is enabled; in SL-free mode it is intentionally suppressed. Missing broker TP is **not** auto-created by default: a manually deleted TP or runner created without `takeProfitOnFill` is treated as a no-broker-TP runner unless `QR_ENABLE_MISSING_TP_REPAIR=1` is explicitly set. Exception: if the position is already in profit and the latest forecast / technical stack no longer justifies carrying the runner into the next session, `tp-rebalance` may add an **insurance TP** near current reward-side price. That is MFE-capture insurance, not a loss-side stop and not permission to cap every runner.
- TP rebalance applies only to existing broker TP orders on trader-owned positions and operator-managed manual/tagless positions. SL repair / tightening applies only to trader-owned positions. Operator-managed manual/tagless positions are never stop-lossed or market-closed by the trader.
- Profitable protected positions can tighten SL to break-even or better.
- **SL-free profit-only break-even/profit-lock stop (2026-05-20; TP-progress gate added 2026-06-03).** `QR_TRADER_DISABLE_SL_REPAIR=1` still means missing SL is intentional and must not be "repaired" on underwater or noise-bound positions. Exception: a trader-owned SL-free position that is already profitable may receive `BREAK_EVEN_STOP` at the original entry price or better only after executable profit pips clear both (i) a fresh market-derived micro-noise envelope: current spread plus fresh M5 ATR and quick M1/M5 realized range from recent candles inside the current M5 management window, and (ii) the TP-progress gate (`QR_PROFIT_BREAK_EVEN_MIN_TP_PROGRESS`, default 60% of current broker TP distance) when a broker TP exists. Stale pair-chart volatility must not be used; if fresh quick volatility is unavailable, the stop is deferred rather than falling back to spread alone. When profit exceeds those gates, the stop is placed behind the current executable price by the same envelope and clamped so it can never be worse than entry. This is not initial SL repair, not trailing SL, and not permission to place loss-side stops; it is a profit-to-flat/profit-lock escape hatch once MFE is large enough that a reversal should not be allowed to turn the trade red, without turning a planned TP trade into a micro-scalp. Manual/tagless positions remain TP-only and must never receive this stop.
- **Bollinger rail continuation handling (2026-05-20; forecast-aware adaptive TP gate added 2026-06-03).** When an already-profitable trader-owned SL-free position has a same-direction BB rail / oscillator setup — e.g. SHORT while M1/M5 price is at the upper Bollinger rail with StochRSI/Williams/MFI overbought, or LONG at the lower rail with oversold evidence — `PositionManager` must not shrink a valid opposite-rail TP just because the micro matrix says HARVEST. If the existing TP already sits at the opposite M5 BB rail, keep that TP and use only the profit-side BE/profit-lock stop. Outside the BB-rail exception, a HARVEST/NARROW contraction of an existing reachable broker TP must prove current market-read MFE risk before replacing the TP: the latest pair forecast must no longer support runner carry and at least two independent technical pressure signals must show exhaustion / fractured agreement / rail risk. Only after that forecast+technical read is present may the adaptive TP-progress gate (`QR_ADAPTIVE_TP_CONTRACTION_MIN_PROGRESS`, defaulting to the same 60% majority-of-target boundary as profit-lock) approve the contraction. A strong same-direction forecast keeps a reachable TP running even when simple progress has cleared. The only early-contraction exception is a stale-wide TP already beyond `QR_TP_REBALANCE_MAX_DISTANCE_ATR × operating ATR`; it may contract when current profit is still smaller than operating ATR, or when the same forecast+technical MFE-risk stack is present. This prevents a +1 pip micro-scalp from replacing a still-reachable +7 pip TP while preserving the stale-target repair that saved trades whose original TP was never realistic. This captures the operator's "BB upper, likely down" read: the rail is continuation evidence for the current SHORT, not a reason to close the short or pull the target prematurely.
- Contradicted trader-owned positions can be closed.
- **Existing SL cannot be widened.**
  - **Existing TP is dynamically rebalanced every cycle** (2026-05-13 update; manual/tagless included 2026-05-15; missing-TP creation disabled by default 2026-05-15; insurance-TP exception 2026-05-15; forecast-harvest contraction added 2026-05-20; stale profitable TP contraction added 2026-05-21; router/verifier enforcement added 2026-05-20). The `tp-rebalance` CLI command (invoked from the trader skeleton between intent pricing and gateway execution) recomputes TP per open trader-owned or operator-managed manual/tagless position using `_market_derived_reward_risk(chart_context) × current_ATR` when a broker TP already exists. `trader-prompt-route` must also run the same calculation in dry-run memory and route to `position_management` whenever an adjustment exists, and `gpt-trader-decision` must reject WAIT / REQUEST_EVIDENCE with `TP_REBALANCE_REQUIRED` while that sidecar is executable. It creates a missing TP only when `QR_ENABLE_MISSING_TP_REPAIR=1` explicitly opts into repair, or when an already-profitable TP-less runner needs insurance because forecast confidence / horizon, next-session timing, price percentile, MTF agreement, 24h range exhaustion, RSI/StochRSI/Williams, Bollinger, or Donchian evidence says the maximum favorable excursion is at risk. A profitable position that already has a broker TP may contract that TP when the latest forecast no longer supports directional runner carry and the technical stack also shows MFE risk; the replacement must be a usable structural HARVEST anchor from `price_action.structural_tp_target`, skipping anchors inside the market safety margin rather than falling back to a guessed pip value. Separate stale-harvest repair is allowed when an attached broker TP is more than `QR_TP_REBALANCE_MAX_DISTANCE_ATR × operating_ATR(M5/M15)` away and the current profit is still smaller than operating ATR, or when a wider technical exhaustion stack is already present; in that case the replacement is a near-market reward-side TP, not a loss-side stop. Once PositionManager or tp-rebalance has set a reachable HARVEST TP inside the operating-ATR cap, later TP passes must not expand it back into a runner merely because unrealized P/L changes before the broker fills that harvest. When the new desired distance differs from the existing TP by ≥ `QR_TP_REBALANCE_HYSTERESIS_PIPS` (default 10), the broker's TP order is replaced. Invariants enforced by `strategy/tp_rebalancer.py`: (a) TP must stay on the correct side of entry (LONG above, SHORT below) so the rebalance cannot lock in a loss, (b) TP must stay ≥ `QR_TP_REBALANCE_MIN_TP_TO_MARKET` pips from current price so it cannot fire on the same tick, (c) TP distance is capped at `QR_TP_REBALANCE_MAX_DISTANCE_ATR × ATR` to bound greed, (d) SL is NEVER touched — SL-free positions remain SL-free, including manual/tagless positions. External positions are skipped. Kill switches: `QR_DISABLE_TP_REBALANCE=1` for the whole pass, `QR_DISABLE_INSURANCE_TP=1` for only missing-TP insurance. User directives 2026-05-13/15/21:「エントリー時だけではなく、途中で伸び縮みできるようにしてほしい。市況によって。それって当たり前だよね？」「基本的には利を伸ばす方向がいいよ。ただ、伸ばすことに固執して、最大含み益の時点で決済できないと意味ない。だからテクニカルをたくさんみなきゃいけない。」「今のショートポジ、逃げておくべきじゃない？プラスのうちに。そういう判断できてる？TP長いんじゃない？」. The original "TP is not moved" rule was the conservative default; dynamic rebalancing replaces it because static entry-time TP ignores the live regime and produced the 5/7-5/13 small-TP-big-DD asymmetry (48 TP fills avg +¥497 vs 42 manual closes avg -¥1,209).
- **New-entry TP execution has two explicit modes** (2026-05-15; stale failed-break initial TP repair added 2026-05-21; technical-HARVEST RR floor clarified 2026-06-17). `intent_generator` always computes a virtual TP for risk/reward validation, but `intent.metadata.attach_take_profit_on_fill` decides whether `LiveOrderGateway` attaches broker `takeProfitOnFill`. `ATTACHED_TECHNICAL_TP` is used for range rotation, failed-break setups, small-wave / low-ADX tape, low TF agreement, exhausted 24h range, or thin liquidity; the TP is anchored on the nearest structural HARVEST target from `price_action.structural_tp_target` when available. If a failed-break attached HARVEST target has no usable structural anchor and the ATR/RR fallback would be more than `QR_HARVEST_TP_MAX_OPERATING_ATR_MULT × operating_ATR(M5)` away, the broker TP must use the minimum acceptable reward-side operating target that clears the technical-HARVEST RR floor (`RiskPolicy.technical_harvest_min_reward_risk`, default 1.0) and spread floor instead of inheriting a distant SL-free virtual risk distance. `RiskEngine` applies that 1.0R floor only to `BREAKOUT_FAILURE` receipts whose metadata declares `opportunity_mode=HARVEST`, `tp_target_intent=HARVEST`, and `tp_execution_mode=ATTACHED_TECHNICAL_TP`; `TREND_CONTINUATION`, forecast runner, spread, loss-cap, forecast/profile, stale-quote, and gateway checks do not get relaxed. Otherwise the ATR/RR virtual target remains visible and risk validation may block it. `RUNNER_NO_BROKER_TP` is reserved for clean TREND/IMPULSE context with ADX ≥ 25 and M15/M30/H1 agreement ≥ 2/3; the broker order omits `takeProfitOnFill`, while the virtual EXTEND target remains in the receipt for TP rebalance, profit partial close, and audit. Global kill switch: `QR_DISABLE_AUTO_TP=1` still omits broker TP for every new entry.
- **S5 bid/ask replay precision geometry is exact-shape only** (2026-06-23; local-TP escape clarified 2026-06-24). A `DAILY_STABLE` S5 bid/ask replay edge may seed a high-turnover `failure_trader` `BREAKOUT_FAILURE` lane only as `TRIGGER_RECEIPT_REQUIRED`, LIMIT-only, `ATTACHED_TECHNICAL_TP`, `tp_target_intent=HARVEST`, with the audited replay TP/SL pips copied into the generated order geometry. Rank-only replay rows remain diagnostics, not executable seeds. The replay proof validates that exact pair/side/forecast-bucket/TP/SL vehicle; candidate generation must not reuse the rule after changing order type, TP distance, SL distance, or HARVEST intent. When capture economics is `NEGATIVE_EXPECTANCY` and local broker `TAKE_PROFIT_ORDER` proof is missing, an exact OANDA campaign-firepower HARVEST vehicle may replace the local-TP proof blocker only if current-risk scaling or the normal equity-derived per-trade cap proves the remaining 5 % floor is reachable within the target trade pace. This is not a broad historical replay waiver: the seeded lane still has to pass current forecast telemetry, strategy profile, spread, profitability acceptance/P0 repair checks, position-guardian active state, risk validation, and gateway broker-truth checks before it can become `LIVE_READY`.
- **Post-TP high-turnover re-entry is evidence-scoped (2026-06-23).** A recent trader-owned `TAKE_PROFIT_MARKET` at a temporary local extreme or a broker `TAKE_PROFIT_ORDER` with positive realized P/L may seed a `post_harvest_trader` `RANGE_ROTATION` re-entry lane only when the same `trade_id` links back to a trader entry lane and the current chart packet still has executable range rails. The seed is always `TRIGGER_RECEIPT_REQUIRED` / LIMIT-only at the pullback rail; it must not become same-pass re-entry, market chase, or permission to ignore forecast, telemetry, strategy profile, spread, month-scale residual loss blocks, or gateway broker-truth validation. This converts the verified positive TP bucket into high-turnover coverage without reopening the market-close loss leak.
- **Profit-side partial close is allowed and gated** (2026-05-15). `profit-partial-close --send --confirm-live` may close part of a trader-owned or operator-managed manual/tagless position only when it is already profitable and has crossed a new ATR-derived milestone. The fraction is market-derived via `dynamic_position_policy.partial_close_fraction`: higher-TF with the position keeps more runner; higher-TF against / choppy tape banks more. The command persists `data/profit_partial_close_state.json` after successful sends so the same trade/milestone cannot fire repeatedly. It never realizes a loss from adverse P/L, never applies to external positions, and does not replace CLOSE Gate A/B for full closes. Kill switch: `QR_DISABLE_PROFIT_PARTIAL_CLOSE=1`.
- **Loss-side adverse partial close is not an implicit live path** (2026-05-29). `adverse-partial-close` is dry-run by default. A live partial reduction requires `--send --confirm-live` plus `QR_LIVE_ENABLED=1`; running the command without these explicit gates must never call `broker.close_trade()`. The scheduled trader does not invoke this module. It remains a diagnostic / operator-only margin relief path and must not be used to bypass CLOSE Gate A/B for full position exits.
- **Profit-side full market take is distinct from loss CLOSE (2026-05-20; temporary top/bottom clarified 2026-06-12; MFE giveback guard added 2026-06-17).** `PositionManager` may emit `TAKE_PROFIT_MARKET` for a trader-owned position only when broker truth says the position is currently profitable and the adaptive TP matrix sees macro reversal against the runner while the trade is in profit. It may also bank a temporary MFE top/bottom when a distribution/rail/local-swing context is present and the M1/M5 execution tape prints a majority of independent rollover readings before the full TP is reached; this is the operator's "一旦の天井で決済して戻りで入り直す" profit-side path, not a loss-cut path. Separately, when recent M1 candles prove executable MFE above current spread/ATR noise, the latest quote is still profitable, a majority of that MFE has already been given back, and fresh reversal evidence is present, `PositionManager` may bank remaining profit even without a clean rail/distribution label; this repairs the execution-timing audit pattern where positive MFE became later red MARKET_ORDER_TRADE_CLOSE. `PositionProtectionGateway` maps this to a full broker close without Gate A/B because it realizes positive P/L and does not cut an underwater thesis. This path is forbidden for manual/tagless or external positions and blocked if the latest broker snapshot is not profitable. Any re-entry must wait for refreshed broker truth and a separate current `LIVE_READY` pullback/retest lane; the profit take itself must not create same-pass re-entry. Loss-side `REVIEW_EXIT` remains governed by Gate A + Gate B below.
  - **Attached-TP progress profit take (2026-06-22; priority clarified 2026-06-22).** If a trader-owned position with an attached broker TP is currently profitable, current executable profit clears both spread-derived and M1 ATR noise, and the position has already captured the high-turnover TP-progress gate (`QR_TP_PROGRESS_PROFIT_TAKE_MIN_PROGRESS`, default 30% of attached TP distance), `PositionManager` may emit `TAKE_PROFIT_MARKET` even before broker TP touch, local top/bottom, or MFE-giveback evidence. This repairs the USD_JPY 472792 failure where a SHORT first cleared executable profit at about 30% TP progress, briefly reached about +3.4 pips / 61% of its attached TP, then reversed to SL because the exit logic waited for the full TP or a later rollover detector. Shallow progress below the gate keeps the broker TP, and post-close re-entry still requires a fresh separate lane. The separate 60% gate still applies to broker-side profit-lock SL and adaptive TP contraction; it must not be reused to delay this spread-paid market-banking path.
  - The TP-progress take must outrank ordinary adaptive `EXTEND_TP`, `HARVEST_TP`, and `NARROW_TP`: once enough attached-TP progress has become executable profit outside current M1/spread noise, stretching or rewriting the target is lower priority than converting that progress into realized campaign P/L. This does not override the specific BB-rail continuation exception or the reachable-TP market-close blocker for macro-reversal closes; it closes the high-turnover path where an aligned runner would otherwise expand/adjust the TP and give back bankable progress before the guardian or next full cycle can react.
- **CLOSE discipline (two-gate, 2026-05-12, `feedback_no_unilateral_close.md`; same-cycle re-entry clarified 2026-06-04; same-cycle re-entry deferred 2026-06-07; Gate B split 2026-06-04).** A `CLOSE` receipt must satisfy Gate A plus the applicable Gate B or be rejected by `gpt-trader-decision`. A `TRADE` receipt must not list `close_trade_ids`: loss-cut first, then end the autotrade cycle as close-only, rerun broker-snapshot / intent generation on the next scheduled cycle, and re-enter only if a fresh `LIVE_READY` lane survives a separate verified `TRADE` receipt. Automation must not immediately re-enter in the same outer autotrade cycle after a close has been sent, staged, or found already satisfied. This prevents "cut and re-enter" from becoming same-cycle recovery chase behavior against just-invalidated context, stale margin, stale quotes, or stale thesis evidence.
  - **Same-direction recovery edge means reprice, not close-and-chase (2026-06-07).** If the current market packet still supports the same direction as the open position, the trader must treat the problem as position geometry, not thesis invalidation: use dynamic TP rebalance, HOLD, profit-side partials, or a separately risk-bounded ADD lane when gateway basket rules allow it. A loss-side `CLOSE` is for a broken thesis. The strongest warning sign is not "same direction after close" by itself; it is closing an underwater runner while the same-direction macro/chart/forecast stack still justifies extending TP, preserving SL-free carry, or repricing the target from current ATR/structure. In that case `CLOSE` should fail Gate A unless explicit hard invalidation evidence also exists.
  - Fresh **hard or explicitly authorized** sidecar Gate A close evidence is priority work, not a protection-mode escape hatch. While `position_thesis_report.json`, `thesis_evolution_report.json`, `position_management.json`, `position_guardian_management.json`, or `forecast_persistence_report.json` carries current `REVIEW_CLOSE` / `RECOMMEND_CLOSE` / `REVIEW_EXIT` evidence for an open trader-owned position and that evidence is hard Gate A or explicit Gate B is present, `gpt-trader-decision` must reject `TRADE`, `WAIT`, `REQUEST_EVIDENCE`, `PROTECT`, and `TIGHTEN_SL` with `POSITION_CLOSE_REQUIRED` until a CLOSE receipt is handled. Soft-only close evidence without Gate B remains advisory for non-CLOSE actions: it may be cited as a reason to monitor, preserve/rebalance TP, or request operator authorization, but it must not freeze TP-managed exposure and block separate current `LIVE_READY` entries on other pairs or horizons. If the trader chooses `CLOSE` from soft evidence, Gate B is still mandatory and `CLOSE_OPERATOR_AUTH_REQUIRED` remains the verifier response unless fresh same-trade HOLD/support evidence exists; then the stronger verifier response is `CLOSE_SAME_DIRECTION_MARKET_SUPPORT`, because the soft close conflicts with current recoverable-position evidence rather than merely lacking a token.
  - **Gate A — market evidence of thesis invalidation.** For each named trade id, either (i) `pair_charts` reports a `BOS_<DIR>` or `CHOCH_<DIR>` against the position side on `M15` or `H4` (parsed from `chart_story`; same structure-event lens trader_brain already uses) AND the structure event is `close_confirmed` (the candle that printed the new swing pivot also closed beyond the broken pivot price — wick-only sweeps marked with the `:wick` suffix do NOT authorize Gate A on their own; added 2026-05-13 after the AUD_JPY M15 BOS_UP@114.146 0.4-pip wick incident), or (ii) the receipt populates `invalidation_price` + `invalidation_tf` AND broker truth clears the level by the anti-wick buffer (default `QR_THESIS_INVALIDATION_BUFFER_PIPS=2.0`; LONG needs `bid <= invalidation_price - buffer`; SHORT needs `ask >= invalidation_price + buffer`) AND operating-timeframe chart/technical confirmation agrees the position thesis is broken, or (iii) a fresh sidecar generated after the current `broker_snapshot.fetched_at_utc` marks the same trade `REVIEW_CLOSE` / `RECOMMEND_CLOSE`: `position_thesis_report.json`, `thesis_evolution_report.json`, or `forecast_persistence_report.json`, or (iv) `position_management.json` or `position_guardian_management.json` marks the same still-open trader-owned trade `REVIEW_EXIT` within the bounded operational carry-forward window before the next PositionManager pass can overwrite it. Forecast-flip and persistence sidecars must require forecast confidence >= `ENTRY_CONFIDENCE_MIN` before producing `RECOMMEND_CLOSE`; weak opposite forecasts are `WEAKENED` / `HOLD` evidence, not Gate A. The `position_thesis` sidecar may also mark a trader-owned underwater legacy/no-invalidation position `REVIEW_CLOSE` when broker truth has moved beyond the entry-price anti-wick buffer and M5/M15/M30/H1 technicals confirm the move against the side; a missing ledger must not make a bad position harder to review than a well-recorded one. However, this no-ledger fallback must degrade to HOLD instead of `REVIEW_CLOSE` when the current position prediction stack still strongly supports that same position direction; early loss-cut is for broken recovery edge, not cutting into a supported recovery merely because the entry ledger was incomplete. A fresh same-trade `position_management` / `position_guardian_management` `HOLD` or `HOLD_PROTECTED` row is also hold-support evidence when it says TP/SL is present, the current thesis is not contradicted enough, latest forecast is unclear/range, or remaining reward is still structurally available; a soft `forecast_persistence` `RECOMMEND_CLOSE` must not override that current technical/HOLD read. The sidecar path is the machine-checkable "plus recovery edge is gone" signal; stale sidecars are ignored. Prose `invalidation` text alone is not enough — the gate requires a machine-checkable signal so the verifier can reproduce the operator's reasoning. Code: `_close_thesis_invalidated` in `gpt_trader.py`; `close_confirmed` flag emitted by `structure._detect_structure_events` and serialised through `chart_reader` as the `:wick` chart_story suffix.
  - **Gate B — standing hard loss-cut authorization OR explicit operator authorization.** Hard Gate A paths satisfy the operator's standing instruction 「妥当な損切りならやっていい」 and do **not** need a fresh 5-minute token: H4 close-confirmed BOS/CHOCH against the side, buffered `invalidation_price` hit with technical confirmation, fresh `thesis_evolution` `BROKEN` / `RECOMMEND_CLOSE`, fresh `position_management` or `position_guardian_management` `REVIEW_EXIT` whose reasons contain structural `loss-cut:` evidence (close-confirmed structural break or multi-TF structural OB break), or fresh `position_thesis` `REVIEW_CLOSE` that records invalidation-hit or structural-break evidence plus M5/M15/M30/H1 technical confirmation for the same trade. M15 close-confirmed BOS/CHOCH remains Gate A evidence, but it is not unattended standing hard authorization unless H4 structure, recorded invalidation, or a hard sidecar also confirms; M15 internal structure or a receipt-level `invalidation_price` must not harden a matching soft `position_thesis` / `position_management` / `position_guardian_management` entry-buffer or unrecorded-invalidation sidecar. If the sidecar says the evidence is adverse-entry-buffer-only, missing-thesis, or entry-thesis-lacks-invalidation, it stays soft and needs explicit Gate B. Softer close evidence still requires explicit operator authorization from one of two operator-controlled channels before a loss-side `CLOSE`: (i) `QR_OPERATOR_CLOSE_OVERRIDE=1` exported in the operator shell, or (ii) a fresh `data/.operator_close_token` file whose mtime is within `OPERATOR_CLOSE_TOKEN_FRESH_SECONDS` (5 minutes; see `gpt_trader.py`). Softer evidence includes `position_thesis` adverse-entry-buffer-only or score-only `REVIEW_CLOSE`, non-structural `position_management` / `position_guardian_management` `REVIEW_EXIT`, `forecast_persistence` `RECOMMEND_CLOSE`, score-only reviews, or any close batch where at least one named trade lacks hard Gate A. Without Gate B, these soft reviews are not a no-trade gate for unrelated `LIVE_READY` entries; they are advisory carry-forward evidence for the next CLOSE decision. Even with Gate B, a soft close that conflicts with fresh same-direction HOLD/support sidecars or same-direction market context is rejected as unsupported by current technical evidence; the operator token authorizes discretion, it does not turn a contradicted soft review into hard thesis invalidation. The receipt's `operator_close_authorized: true` field is **advisory only** and is **not accepted as authorization**.
- The close execution layer must only call broker close for GPT summaries with `status=ACCEPTED`, `allowed=true`, and `action=CLOSE`. Rejected summaries, WAIT / TRADE / CANCEL / PROTECT receipts, and any `TRADE` receipt carrying `close_trade_ids` must not reach the broker close path. Even after Gate A/B acceptance, `PositionProtectionGateway` must re-check the latest broker-snapshot quote before sending a market CLOSE: missing quotes, missing normal-spread baselines, or current spread wider than the broker-spec `NORMAL_SPREAD_PIPS × RiskPolicy.max_spread_multiple` cap block the close with `POSITION_CLOSE_*` issues unless the operator explicitly exports `QR_POSITION_CLOSE_SPREAD_OVERRIDE=1`. Low-level `OandaExecutionClient.close_trade()` must raise before network I/O; the gateway and explicitly gated partial-close CLIs call `close_trade_with_provenance()` instead, and persist that provenance in the position-execution receipt. This is the exit-side mirror of the entry spread gate and prevents structurally valid CLOSE receipts from becoming forced fills in rollover / thin-liquidity spreads (2026-06-11 OANDA audit: GBP_CHF loss close paid 591 JPY half-spread cost on a ~30-pip spread).
- `gpt-trader-decision` must persist `close_gate_evidence[]` for every named CLOSE trade id, accepted or rejected: trade id, pair/side, unrealized P/L, loss-side flag, Gate A verdict and reason, standing-vs-explicit Gate B state, profitability-P0 citation state, timing-audit citation state, hard-timing requirement, and any same-direction support conflict. `verification-ledger-audit` must normalize those rows into `verification_observations` with `check_name=close_gate_evidence`, so a future loss-side `MARKET_ORDER_TRADE_CLOSE` can be traced by SQL without manually re-reading every decision packet. Missing durable close-gate evidence is an audit defect, not proof that the close was justified.
- Broker-close reconciliation is not proof of close discipline by itself (2026-06-22). `execution-ledger-sync` may reconcile an OANDA `TRADE_CLOSE` against a trader entry lane after the fact, but a loss-side `MARKET_ORDER_TRADE_CLOSE` is verified only when the ledger also contains a durable `GATEWAY_GPT_CLOSE_ACCEPTED` and/or `GATEWAY_TRADE_CLOSE_SENT` receipt for the same close path. `profitability-acceptance` must classify trader-entry broker-close reconciliation without those receipts as `GATEWAY_TRADE_CLOSE_RECONCILED_UNVERIFIED` and keep a P0 `UNVERIFIED_LOSS_SIDE_MARKET_CLOSE_RECONCILED` active inside the recent leak window. Self-improvement loss-containment math must not count an unverified reconciled close as a proved GPT/gateway loss-cut recovery. This prevents post-hoc attribution from hiding the exact path that has been erasing TP-captured edge.

## 13. Engineering Conventions

### Package manager
- None. Use system Python with `PYTHONPATH=src`.
- Test with `PYTHONPATH=src python3 -m unittest discover -s tests -v`.

### Commit attribution
AI commits MUST include the line for the operator who authored the change:

- Codex commits:
  ```
  Co-Authored-By: Codex <noreply@openai.com>
  ```
- Claude commits:
  ```
  Co-Authored-By: Claude <noreply@anthropic.com>
  ```

If both contributed in the same commit, include both lines.

### Always keep commits current
- The working tree must be **committed before ending a session**, before switching operator (Codex ↔ Claude), and before any scheduled-task cycle that may run live (`autotrade-cycle --send`).
- Commit logical units as they finish — do not let unrelated changes pile up across sessions. Untracked or uncommitted work invalidates the audit trail and risks the next operator inheriting an unknown state.
- Receipts and reports written into `data/` and `docs/*_report.md` are runtime artifacts; commit them only when they are part of a deliberate snapshot, not on every cycle (they are usually `.gitignore`-d under `data/`).
- Before starting work, run `git status` / `git log -1` to confirm the tree is clean and the previous operator's commit is the latest. If it is not, either commit the leftover changes (with the correct operator attribution) or ask the user before discarding.
- Never `--amend` a published commit, never force-push, and never skip hooks (`--no-verify`). Stack a new commit instead.

### Acceptance bar
- Add a **failing regression test** for known legacy failure modes.
- Add a **passing test** for new behavior.
- Document the live-risk failure mode for accepted rebuild components.
- Preserve dry-run receipt paths for new execution features.
- **No live side effect unless explicitly enabled.**
- Never promise market-guaranteed returns. Encode the 5 % guarantee as a system obligation, audit ledger, incident trigger, and bounded, testable execution behavior.
